## Findings

### Medium/low - Non-explicit selected slots can still drive top/map state while detail rows stay in summary mode

This is not a hard blocker for the specific v013 finding, but it is the main behavior to validate on localhost.

`FerdalagidClient.tsx` now gates `RoutePointRow` active mode on `userExplicitSlot`:

- `app/auth-mvp/vedrid/FerdalagidClient.tsx:952-957`

That correctly restores default rich metrics in `Allir spapunktarnir a leidinni` after a fresh calculation.

However, other slot-specific UI still uses `selectedHeatmapIdx` / `selectedReturnHeatmapIdx` even when `userExplicitSlot` is `false`:

- Auto-select first outbound slot after a new result: `app/auth-mvp/vedrid/FerdalagidClient.tsx:386-392`
- Filter auto-select can change selected slot directly: `app/auth-mvp/vedrid/FerdalagidClient.tsx:396-442`
- Highlighted issue, selected point statuses and active candidate still derive from selected indices: `app/auth-mvp/vedrid/FerdalagidClient.tsx:491-512`
- Top card status sentence uses `activeOutboundCandidate`: `app/auth-mvp/vedrid/FerdalagidClient.tsx:809-846`
- Map receives slot-specific statuses/candidate without the `userExplicitSlot` gate: `app/auth-mvp/vedrid/FerdalagidClient.tsx:897-913`
- Map colors/counts/ETA chips use selected candidate data whenever `selectedCandidatePointStatuses` is defined: `components/weather/TravelAuditMap.tsx:311-355` and `components/weather/TravelAuditMap.tsx:392-409`

Impact:

- After initial calculation, or after filter auto-selection, the heatmap/top card/map can behave as if a slot is selected.
- At the same time, the detail list intentionally stays in default `summaryForWindow` mode until Stebbi explicitly clicks a slot.
- This may be an acceptable v014 compromise, but it can still feel like two parts of the same screen are explaining different time contexts.

Recommendation:

- Accept v014 if localhost testing feels clear enough.
- If it still feels confusing, next safest fix is to split state into two concepts:
  - `visualSelectedSlotIdx` for heatmap outline/top/map behavior.
  - `explicitDetailSlotIdx` for `Allir spapunktarnir a leidinni`.
- Alternative: pass the `userExplicitSlot` gate to the top/map too, but that may change current default map/top behavior more broadly.

### Low - No targeted regression test for explicit-vs-auto slot behavior

The full test suite is green, but I did not find a focused test that locks down the new `userExplicitSlot` behavior. This is a UI state bug class that can regress easily because auto-selection, filters, map selection and detail rows all share adjacent state.

Recommendation for a later cleanup:

- Extract a small display-model helper for point-row mode, or add a component test that covers:
  - fresh route result with auto-selected first slot keeps `summaryForWindow` metrics visible;
  - explicit no-data slot suppresses stale weather metrics and shows the no-data detail copy;
  - deselect returns to summary metrics.
