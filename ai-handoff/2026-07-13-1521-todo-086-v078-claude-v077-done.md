### P1 — Upsert-first, delete-stale-after

`lib/weather/providers/vedurstofan.server.ts`

Old behavior: delete all existing rows → insert new rows. Insert failure left station empty.

New behavior:
1. Upsert new forecast rows (`onConflict: 'station_id,forecast_time'`)
2. If upsert fails → error++, continue. Existing rows preserved.
3. If upsert succeeds → delete stale rows (`.eq('station_id', id).lt('fetched_at', payload.fetchedAtIso)`)
4. Delete failure is non-fatal (logged, not counted as error) — stale extra row is preferable to empty station.

### P2a — Cache scan failure is now visible

Old: `{ projected: 0, skipped: 0, errors: 0, runId: null }` — indistinguishable from "nothing to project".

New: `{ projected: 0, skipped: 0, errors: 1, runId }` — `errors: 1` and `runId` returned from the written run record.

Applies to both `error || !data` case and exception case.

### P2b — Stricter validation

1. **Cache key mismatch**: extract station ID from `cache_key` suffix, compare to `payload.stationId`. Mismatch → `skipped++`, no write.
2. **Invalid forecast rows**: filter `payload.forecasts` to those with non-empty `ftimeIso`. If zero valid remain → `skipped++`, no write. Individual bad rows dropped silently, rest projected.
