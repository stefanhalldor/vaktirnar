## Findings

### Major 1 - Claude is right: do Phase 0 before provider refactor

Claude's main correction is valid. We should not start a MET provider wrapper / Phase 1 abstraction until Phase 0 has answered the unknowns around Veðurstofa data shape, station coverage, endpoint choice, time resolution and cache shape.

Reason: if Veðurstofa is station-batched rather than point-coordinate-based, the provider interface we design before checking the actual source can easily be wrong.

Recommendation: next step should be **Phase 0 investigation only**.

### Major 2 - Veðurstofa must be batch-by-station, not per route point

Claude is right that current `route.ts` fetches MET in a per-point pattern:

```ts
Promise.allSettled(weatherPoints.map((pt) => fetchForecast(pt.lat, pt.lon)))
```

That pattern must not be copied for Veðurstofan. For Veðurstofa we need:

1. map route points to station IDs;
2. dedupe unique station IDs;
3. fetch each station once;
4. attach station result + confidence metadata back to route points.

This points toward a dedicated route-level provider/batcher, not a simple `fetchForecast(lat, lon)` clone.

### Major 3 - `weather_cache.response_body` is `jsonb`

Confirmed in `sql/67_weather_cache.sql`:

```sql
response_body jsonb NOT NULL
```

This does not necessarily force a new SQL migration. PostgreSQL `jsonb` can store a JSON string, so raw XML could be stored as a JSON string value if Supabase serializes it that way. But it needs to be explicit and tested.

Phase 0 should verify:

- Can Supabase upsert a raw XML string into `response_body jsonb` through the JS client?
- Does reading it back preserve the exact string?
- Do we want to store XML string, parsed JSON, or both?

If exact raw XML storage becomes awkward, options are:

- parse XML immediately and cache parsed normalized JSON only;
- store `{ rawXml: string, parsedAtIso, sourceMeta }` as JSONB;
- add a text column or separate cache table later, but only if needed.

Do not write SQL yet.

### Minor 1 - Phase 2 and Phase 3 should be merged/resequenced

Claude's critique is fair: a Veðurstofa provider cannot be meaningfully tested without station mapping.

Better phase split:

- Phase 2a: station list + mapping skeleton;
- Phase 2b: fetch/parser/cache per station;
- Phase 2c: route-level batcher that maps route points to station forecasts.

### Minor 2 - "server-side spike script" is still an implementation

Claude suggests `scripts/vedurstofan-spike.ts`. That is a reasonable tool, but it is a file creation/code change.

Per workflow, Claude should not create it unless Stebbi gives explicit implementation permission. A read-only Phase 0 can start by reading docs and migrations, but as soon as a script is created or run against external endpoints, that needs a clear go-ahead.

### Minor 3 - Endpoint choice should include `api.vedur.is`, not only XML

Claude correctly calls out that we should not assume the old XML service if `api.vedur.is` has a better current endpoint.

Phase 0 output should explicitly compare:

- old XML/CSV `xmlweather.vedur.is`;
- any documented/current `api.vedur.is` endpoint;
- data fields, rate behavior, time resolution and station metadata.
