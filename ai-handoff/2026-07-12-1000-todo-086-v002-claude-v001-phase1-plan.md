## Forsendur staðfestar

Ég las kóðann sem Codex vísar í. Niðurstaðan er:

### `metno.server.ts` (71-133 línu)

- `fetchForecast(lat, lon)` er eina public API-ið. Það skilar `HourPoint[]`.
- Cache key: `metno:locationforecast:2.0:compact:{lat}:{lon}` (3dp rounded).
- Cache table í Supabase: `weather_cache`, dálkar: `cache_key`, `response_body`, `expires_at`, `last_modified`, `fetched_at`, `updated_at`.
- Stale fallback á allar HTTP villur, 403, 429, network error.
- Expires-header parsed, default 1 klst.

### `forecast.ts` (parseMetnoForecast)

Staðfest það sem Codex sagði um "0 = vantar":

```ts
airTemperatureC: d.air_temperature ?? 0,
windSpeedMs: d.wind_speed ?? 0,
windGustMs: d.wind_speed_of_gust ?? d.wind_speed ?? 0,  // fallback to wind if no gust
windFromDegrees: d.wind_from_direction ?? 0,
```

Einnig: línur 33-38 í filter-stigi: entries án `wind_speed` eða án `next_1_hours`/`next_6_hours` eru þegar dropp-aðar. Þetta er "no data = filter out" í MET kóðanum, en við Veðurstofuna þarf að meðhöndla missing á öðrum tíma í pipeline.

### `routeSampling.ts`

Staðfest: `forecastLat`/`forecastLon` eru `roundCoord()` (3dp). Þetta er MET-sértækt. Veðurstofan þarf station ID, ekki rounded coordinate.

### `route.ts`

Lykilatriðið sem Codex vantar: **`fetchForecast` er kallaður á hvern route point í parallel** (lína 243):

```ts
Promise.allSettled(weatherPoints.map((pt) => fetchForecast(pt.lat, pt.lon)))
```

Þetta þýðir að ef við bætum Veðurstofu provider við á sama stigi, þá mun `weatherPoints.map(...)` verða tvöfaldað. Ef leiðin er 80 punktar og við sækjum bæði MET og Veðurstofuna = 160 requests í einu. Veðurstofan samþykkir ekki það án cache.

Vandamálið við station-based provider: Veðurstofan á **ekki** einn endpoint per point eins og MET. Við þurfum að:
1. Finna unique station IDs fyrir alla route points fyrst.
2. Sækja hverja stöð einu sinni.
3. Map-a route points á station result.

Þetta krefst fundamentally öðruvísi fetch-patterns en núverandi `fetchForecast()`.

### `assessment.ts`

Staðfest domain seam: `assessRouteLeg()` tekur `TravelPointForecast[]` og thresholds. Kjarnamynstrið er hreint og provider-agnostic eins og það stendur.

## Rýni á Codex Phase plan

### Sterkt

- Phase 0 (read-only spike) áður en nokkuð annað er gert: þetta er rétt. Við vitum ekki hvort Veðurstofu station gögn séu nógu þétt á þeim leiðum sem við sýnum.
- `ForecastProviderResult` wrapper utan um `HourPoint[]`: þetta verndar `assessment.ts` domain seam.
- Cache sem öryggisskilyrði, ekki optimization: mjög rétt.
- Station confidence metadata: nauðsynlegt.

### Þarf leiðréttingu eða frekari skýringar

#### 1. Phase order vandamál: Phase 2 á undan Phase 3

Codex leggur til að búa til `vedurstofanProvider.server.ts` í Phase 2, en station mapping kemur ekki fyrr en Phase 3.

Vandinn: provider getur ekki skilað gögnum án station mapping. Þú getur ekki prófað provider án þess að vita hvaða station á að sækja. Phase 2 og 3 eru í raun eitt verk: "fetch + parse + map stations."

Tillaga: sameina Phase 2 og 3, eða kalla þá "Phase 2a" (station list og mapping skeleton) og "Phase 2b" (parser + fetch).

#### 2. `fetchForecast()` compatibility wrapper er flóknari en Codex gefur til kynna

Codex segir í Phase 1: "Halda `fetchForecast()` sem compatibility wrapper svo núverandi flow brotni ekki."

En `fetchForecast()` er kallaður **í `route.ts`** per route point, ekki per provider. Ef við wrap-um `fetchForecast` í `metnoProvider.server.ts`, þá er það óháð því hvort Veðurstofa provider er til. MET provider wrapping í Phase 1 breytir engu í `route.ts` og engum visible hegðun.

Þetta er í lagi, en Stebbi þarf að skilja: Phase 1 MET wrapping er invisible refactor. Engin breyting á output, engin ný test coverage á `route.ts` level, bara innskot á milli. Það er not-bad, en það er ekki zero risk. Tests þurfa að sanna að MET wrapper skilar nákvæmlega sömu gildum.

#### 3. Multi-source fetch pattern í `route.ts` þarf að ákveða núna

`route.ts` kallar `fetchForecast` per point í `Promise.allSettled`. Þegar Veðustofa kemur inn, hvernig breytist þetta?

Kostur A: `route.ts` kallar `fetchForecastFromAllProviders(pt)` sem skilar `ForecastProviderResult[]`.

Kostur B: `route.ts` veit af báðum providers og kallar þá í parallel per point.

Kostur C: Veðustofan er station-batched, þannig að `route.ts` fær sérstaka `fetchVedurstofanForRoute(weatherPoints)` batch-fall.

Codex Phase plan sér ekki um þetta. Við þurfum að ákveða Kost A/B/C áður en Phase 1 MET wrapping byrjar, annars verður interface ósamhæft.

Mín tillaga: **Kost C** fyrir Veðurstofuna, af því station fetch er batch by nature. MET provider heldur áfram per-point. `route.ts` kallar þá hvort um sig á mismunandi tíma, samanber flag.

#### 4. `weather_cache` tafla er shared

Veðurstofa cache og MET cache deila sömu `weather_cache` töflu. `cache_key` er string og er source-specific ef við fara í `vedurstofan:xml:forec:{stationId}:{timeStep}`. Þetta virkar og krefst engrar SQL migration ef schema er flexibelt.

Staðfest: schema hefur `cache_key text unique`, `response_body jsonb/text`, `expires_at`, `last_modified`. XML body þarf að geymast sem text, ekki JSONB. Þarf að skoða hvort `weather_cache.response_body` er `jsonb` eða `text`.

**Óvissa:** ef `response_body` er `jsonb` column, getur Supabase geymt raw XML string í honum? Supabase geymist JSON, ekki XML, í JSONB. Þarf að skoða DDL eða migration skrána. Ef XML þarf að geymast gæti þurft annað approach (base64, eða sér text column).

#### 5. Phase 0 spike þarf að koma á undan öllum öðrum áföngum

Codex segir þetta, en listinn er Phase 0, Phase 1, Phase 2... og gefur til kynna að við getum hafið Phase 1 strax. Mín skoðun: við á ekki að byrja Phase 1 (MET wrapping) fyrr en Phase 0 spike hefur svarað þessum spurningum:

1. Hvaða endpoint: `forec` API eða annað?
2. Hvert er raunverulegt station coverage á algengum ferðaleiðum (1, 41, 48, 51)?
3. Eru stöðvar nógu þéttar til að gefast upp á MET-uppfyllinguna?
4. Er time resolution 1h, 3h eða 6h?
5. Er response XML eða JSON (ný API vs gamla)?
6. Er `response_body` í `weather_cache` JSON eða text?

Þar til við vitum þetta eru Phase 1-6 ofur-óvissar.

## Óvissa / þarf að staðfesta

- `weather_cache.response_body` column type - JSON vs text - áhrif á XML caching.
- Raunverulegt station coverage á íslenskum vegum.
- Veðurstofu rate-limit policy (docs eru óljósar um nákvæmar tölur).
- Time resolution per station (gæti verið breytileg eftir stöð/líkan).
- Hvort `api.vedur.is` (nýja REST API) sé nógu þroskað til að nota í stað XML.
