## Findings

### High - Route memory can delete Veðurstofan station rows when the Veðurstofan layer is feature-gated off

`app/api/teskeid/weather/travel/route.ts:267` sets `vedurstofanMatches = []` when `layerEnabled` is false, but the route-memory write still sends `providersEvaluated: ['vedurstofan']` at `app/api/teskeid/weather/travel/route.ts:486`. `recordRouteMemory()` treats every provider in `providersEvaluated` as evaluated and deletes existing station rows for that provider before inserting the fresh rows, including the 0-row case (`lib/iceland-routes/routeMemory.server.ts:115`).

Impact: if `WEATHER_PROVIDER_VEDURSTOFAN_ACCESS_REQUIRED=true` and a user without Veðurstofan access computes a route that already has route-memory station rows, the request can clear the stored Veðurstofan stations for that route. That can make `/vedrid` lose exact route-memory filtering for that route. This is a production-data risk, even though the helper is best-effort.

Suggested fix before release: build `providersEvaluated` from actual provider evaluation:

```ts
const providersEvaluated: Array<'vedurstofan' | 'vegagerdin'> = []
if (layerEnabled) providersEvaluated.push('vedurstofan')
if (vegagerdinAvailable) providersEvaluated.push('vegagerdin')
```

Alternative if route-memory should always keep recording Veðurstofan even when the display layer is hidden: split "display layer enabled" from "provider station matching for memory", and keep memory matching independent of the user-facing feature gate.

### High - Status pills still filter only met.no route points, not the new primary Veðurstofan station layer

`components/weather/RoadMapPrototypeMap.tsx:209` and `components/weather/RoadMapPrototypeMap.tsx:365` apply status filters only to `travel-bridge-weather-points`. The new `vedurstofan-route-stations` layer added at `components/weather/RoadMapPrototypeMap.tsx:714` is never classified with `windDisplayStatus` and never filtered by `WindStatusFilterPills` at `components/weather/RoadMapPrototypeMap.tsx:1629`.

Impact: the user can click the same "Innan marka / Óþægilegt / Hættulegt" pills, but the visually prominent teal Veðurstofan dots stay on the map regardless. Since v276 intentionally makes Veðurstofan more primary, this makes the filter semantics misleading.

Suggested fix before treating this as a releaseable UX: classify each Veðurstofan station with the same wind-threshold logic and apply the same filter to both layers. If that is too large for the next patch, hide or relabel the pills on the prototype so they are clearly "met.no grunnpunktar" only.

### Medium - Veðurstofan popup wind uses the first forecast row, not the ETA-nearest row

`components/weather/RoadMapPrototypeMap.tsx:733` uses `p.forecastRows[0]?.windSpeedMs`. For a long route, station weather should represent the expected arrival time at that station, not simply the first row returned by the server window.

Impact: the popup can show the wrong time slice even though the backend fetched a wider ETA window. This weakens the "Veðurstofan-first route weather" story.

Suggested fix: compute the ETA-nearest forecast row on the server, using station `routeFraction`, route duration and departure time, then expose it as e.g. `selectedForecastRow` in `VedurstofanTravelLayerPoint`. The client should render that row and use it for classification.

### Medium - The route result still looks like Veðurstofan drove the answer, but the actual route assessment is still met.no/Yr

`app/api/teskeid/weather/travel/route.ts:316` still calls `checkTravelWeather()` with `pointForecasts` from met.no route samples. v276 returns `vedurstofanLayer` as an extra client layer at `app/api/teskeid/weather/travel/route.ts:512`, but the main route status, counts and text are still based on the met.no baseline.

Impact: the summary can say something like `80 veðurpunktar` plus `X Veðurstofan stöðvar`, while the visible map foregrounds Veðurstofan. A tester can reasonably infer that Veðurstofan is driving the safety answer, when it is currently an overlay.

Suggested fix: either label the current answer as "grunnmat" / "met.no/Yr" until provider blending is implemented, or move the status calculation into a shared provider-aware path before calling it "Veðurstofan-first".

### Low - Repeated clear/recalculate cycles can accumulate MapLibre layer event handlers

`handleClearRoute()` removes `vedurstofan-route-stations` layer/source at `components/weather/RoadMapPrototypeMap.tsx:478`, but the click/mouseenter/mouseleave handlers registered at `components/weather/RoadMapPrototypeMap.tsx:764` are not removed. MapLibre layer handlers can persist across layer removal and fire multiple times after the layer is recreated.

Impact: repeated route calculations may duplicate popups or cursor handlers. This is probably minor in early prototype use, but easy to harden.

Suggested fix: keep named handler functions and call `map.off(...)` before removing the layer, or register handlers once in the map init effect and leave them stable.

### Low - The client trusts the extra `vedurstofanLayer` response shape too much

`components/weather/RoadMapPrototypeMap.tsx:890` casts the raw response extra field directly to `VedurstofanTravelLayer`. `renderVedurstofanStations()` then calls `(layer?.points ?? []).filter(...)`. If `points` is present but not an array, this would throw.

Impact: low, because the API is first-party, but this is an experimental surface and a cheap defensive guard.

Suggested fix: use `const points = Array.isArray(layer?.points) ? layer.points : []`, and require `Number.isFinite(p.lat)` / `Number.isFinite(p.lon)`.

## What Looks Good

- The route geometry fix is directionally correct. Using `routeGeometry.providerMatchingPoints ?? routeGeometry.points` for provider matching and `auditPolylinePoints` should remove the worst straight-line route rendering caused by the 80-point met.no sample.
- `lib/weather/travel.ts` already caps static-map audit URLs through `buildAuditMapUrl()`, so passing dense audit geometry should not explode Google Static Maps URL length.
- Veðurstofan layer fetch is fail-open from the route user experience perspective: if station product lookup fails, the baseline result still returns.
- `npm run type-check` passed after the reviewed changes.

## Recommendation

I would not present v276 as "Veðurstofan-first route weather" yet. It is fine as a feature-flagged prototype after fixing the route-memory cleanup risk. Before broader release or serious user testing, I would patch:

1. route-memory `providersEvaluated` so hidden/unauthorized Veðurstofan evaluation cannot clear stored stations;
2. status classification/filtering for Veðurstofan station markers;
3. ETA-nearest Veðurstofan row selection for popup/status.

The route geometry part itself is the strongest part of v276.
