### P1 — Stale route line when origin changes with destination set

`components/weather/RouteSelectionStep.tsx` was already rewritten (v088) with
4-effect architecture:

- Effect 1: map init (mount only)
- Effect 2: origin marker only (`[origin?.lat, origin?.lon, mapLoaded]`)
- Effect 3: destination marker only (`[destination?.lat, destination?.lon, mapLoaded]`)
- Effect 4: route line + bounds — clears line unconditionally, redraws when both
  points exist (`[origin?.lat, origin?.lon, destination?.lat, destination?.lon, mapLoaded]`)

Changing origin while destination is set now triggers Effect 4, which clears the
old A→B line and draws C→B.

### P1 — Heatmap selection not connected to audit map

Lifted heatmap selection state to `FerdalagidClient`:

```typescript
const [selectedHeatmapIdx, setSelectedHeatmapIdx] = useState<number | null>(null)
```

`DepartureHeatmap` is now fully controlled — `selectedIdx` and `onSelectIdx` are
passed from the parent.

Derive `heatmapHighlightedIssue`:

```typescript
const heatmapHighlightedIssue =
  selectedHeatmapIdx !== null && candidates[selectedHeatmapIdx]
    ? candidateToIssue(candidates[selectedHeatmapIdx])
    : result?.travelPlan?.highlightedIssue
```

This is passed as `highlightedIssue` to `TravelAuditMap` (replacing the old
`result.travelPlan?.highlightedIssue`).

`TravelAuditMap` gained a new effect that responds to `highlightedIssue` changes
from the parent:

```typescript
useEffect(() => {
  if (!highlightedIssue?.lat || !highlightedIssue?.lon) return
  const idx = weatherPoints.findIndex(
    p => p.lat === highlightedIssue.lat && p.lon === highlightedIssue.lon,
  )
  if (idx >= 0) setSelectedIndex(idx)
}, [highlightedIssue, weatherPoints])
```

Tapping a heatmap slot now moves the map's selected marker, updates the
`PointDetailsPanel`, and shows the matching metric/threshold for that exact
departure candidate. Tapping the same slot again deselects it (restores default
highlighted issue). Deselecting returns the map to the original result's
highlighted issue.

`selectedHeatmapIdx` is reset in both `handleSubmit` and `startOver`.

### P1 — Reverse geocode BFF public with no kill switch or rate limit

`app/api/place/reverse-geocode/route.ts` now has:

1. **Kill switch**: `ENABLE_REVERSE_GEOCODE=false` in env → 503 immediately.
2. **Per-IP rate limit**: max 20 requests per 60 seconds (in-memory, best-effort).
   Returns 429 when exceeded.

`.env.example` updated with `ENABLE_REVERSE_GEOCODE=` documentation comment.

Note: in-memory rate limit does not persist across serverless instances. For
production traffic, upgrade to Redis/Upstash or rely on edge rate limiting if
needed. The kill switch is unconditional and safe.
