## Findings

### Medium - Route-pair hashes are written with an empty HMAC secret if `USAGE_EVENT_SECRET` is missing

`lib/teskeid/usage.server.ts` currently falls back to an empty string:

- `lib/teskeid/usage.server.ts:60` says the hash is not cryptographically secure until the secret is set.
- `lib/teskeid/usage.server.ts:67` uses `process.env.USAGE_EVENT_SECRET ?? ''`.
- Both weather endpoints always call `routePairFingerprint(...)` before recording metadata:
  - `app/api/teskeid/weather/travel/routes/route.ts:94`, `:105`, `:119`
  - `app/api/teskeid/weather/travel/route.ts:187`, `:201`, `:211`, `:242`, `:269`

This is risky because route-pair hashes are meant to be the privacy boundary for distinct route pairs. If production code is deployed and migration 71 is run before the env var is configured, the database starts accumulating deterministic unsalted hashes. Even if the admin API does not expose hashes, the table itself contains weaker location-derived identifiers.

Recommendation: do not store `routePairHash` unless a real secret exists. Safer options:

- make `routePairFingerprint` return `null`/`undefined` when `USAGE_EVENT_SECRET` is missing, and omit `routePairHash` from metadata;
- or make usage recording for route-pair distinct counts disabled until the secret exists, while still recording total events;
- add a test that verifies no route-pair hash is inserted when the secret is missing.

Do not make user-facing weather requests fail because the usage secret is absent.

### Medium - The "migration not run" empty state does not actually show when the table is missing

The handoff says the admin UI shows an empty state with a migration note if migration 71 has not been run. In the current code, a missing table causes the admin API to return 500:

- `app/api/admin/teskeid-usage/route.ts:80-85` returns `{ error: 'Query failed' }` with status 500 on any Supabase query error.
- `app/(admin)/admin/page.tsx:443-448` discards non-OK usage responses and leaves `usage` unchanged/null.
- `app/(admin)/admin/page.tsx:276-299` renders the migration note only when `usage` is non-null and `summary.total_events === 0`.

So if migration 71 is not run, the section will be hidden, not shown with the migration note. This also means the stated localhost check is misleading.

Recommendation: either:

- have `/api/admin/teskeid-usage` detect the missing relation error and return a safe zero-state payload with a `migration_missing` flag, or
- track usage fetch errors in admin UI and render a calm "migration/API unavailable" state.

Keep true unexpected DB errors distinguishable in logs. Do not expose raw Supabase error details to the browser.

### Medium - SQL migration 71 should be wrapped in a transaction before Stebbi runs it

`sql/71_teskeid_usage_events.sql` creates the table, indexes, RLS and grants, but has no `BEGIN;` / `COMMIT;` wrapper:

- `sql/71_teskeid_usage_events.sql:8-47`

Most recent security-sensitive migrations in this project are careful about RLS/grants and often use transaction boundaries for related changes. If Stebbi pastes this into Supabase SQL editor and a later statement fails, a partial schema/grant/RLS state is harder to reason about.

Recommendation: wrap the migration in `BEGIN; ... COMMIT;` before running it. This is especially appropriate because the table, indexes, RLS and grants are one unit.

### Low - New SQL migration is not covered by static SQL migration tests

The implementation adds `sql/71_teskeid_usage_events.sql`, but the existing SQL static tests do not appear to cover it. `rg` shows the file itself but no `sql71`/`teskeid_usage_events` assertions under `lib/__tests__` except runtime helper/API tests.

Recommendation: add a focused static SQL test for migration 71 covering at least:

- table exists;
- RLS is enabled;
- `REVOKE ALL ... FROM PUBLIC, anon, authenticated`;
- `GRANT SELECT, INSERT ... TO service_role`;
- no anon/authenticated grants;
- metadata object check exists;
- transaction wrapper exists.

### Low - Admin UI copy has a typo and a mixed-language label

In `app/(admin)/admin/page.tsx`:

- `:299` says `Engar virknifarar í þessum tímaramm.`. This should likely be `Engin virknigögn á þessu tímabili.` or similar.
- `:286` and `:335` use `Distinct leiðapör`. Product/admin Icelandic would probably read better as `Sérstök leiðapör` or `Ólík leiðapör`.

Not a blocker, but worth fixing while Claude Code is already touching the usage section.

## What looks good

- The usage table is service-role only in shape: RLS enabled, grants revoked from public roles, explicit `SELECT, INSERT` to service_role only.
- Weather instrumentation is after auth and feature access checks.
- Usage insert failures are swallowed and should not break user-facing Veðrið.
- Admin API response is aggregated and does not expose raw rows, `user_id` or route-pair hash lists.
- Tests cover helper sanitization, admin API auth/aggregation, and weather route-options usage events.
