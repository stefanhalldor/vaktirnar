## Tvær athuganir sem þarf að ákveða

### 1. `/travel/route.ts` — public final forecast: gera samtímis eða seinna?

v010 lætur þetta opið. Vandinn ef við skiptum í tvo hluta:

- Dag 1: public route calculations teljast → `weather_route_options_calculated` fær public talningu
- Dag 2+ (síðar): public final forecast teljast → `weather_final_forecast_completed` fær public talningu

Á meðan eru tvær útkomur:
- Conversion rate (leiðir → niðurstaða) = 0% fyrir public í admin
- Eða conversion rate er ekki sýnd — en þá vantar Stebba gagnlegar upplýsingar

**Tillaga**: Gera báðar breytingarnar í sama PR. Það er lítið viðbótarverk (sama pattern, sama `userId: null` breyting á 2-3 línur í `/travel/route.ts`) og forðast villa í admin í nokkra daga. Þetta er einungis raunhæft ef v010 er allt gert í einu.

Ef Stebbi vill senda það í tveimur PRum: þá á admin UI að sýna conversion **aðeins** sem `Innskráðir` þangað til public final forecast er líka skráð — með skýrri athugasemd í UI.

### 2. `actor` bætt við existing authenticated events — breytir existing metadata

Handoffinn segir að authenticated events fái líka `actor: 'authenticated'` í metadata. Þetta þýðir að eldri authenticated events (sem eru þegar í DB án `actor`) og nýrri events hafa mismunandi metadata shape.

Admin aggregation á að notast við:
> "Prefer using `metadata.actor` when present, but keep null-user fallback for compatibility."

Þetta er rétt og þarf að vera nákvæmlega útfært svo eldri events teljist enn sem `authenticated` jafnvel þótt þeir hafi ekki `actor` field.

**`weather_route_calculations` vs `weather_route_options_calculated`**



Eða: halda `weather_route_options_calculated` sem total og búa til derived counts í admin API úr `metadata.actor` (þ.e. telja per actor, ekki geyma sem sér event type). Þetta er þægilegri lausn og krefst ekki nýrra event names.
