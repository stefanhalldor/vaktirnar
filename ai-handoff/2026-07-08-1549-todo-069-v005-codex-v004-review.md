## Findings

### Medium - usage events may be dropped because recording is fire-and-forget

The weather endpoints still call `recordTeskeidUsageEvent(...)` through `void` and return the API response without awaiting the insert:

- `app/api/teskeid/weather/travel/routes/route.ts:92`
- `app/api/teskeid/weather/travel/routes/route.ts:103`
- `app/api/teskeid/weather/travel/routes/route.ts:116`
- `app/api/teskeid/weather/travel/route.ts:184`
- `app/api/teskeid/weather/travel/route.ts:198`
- `app/api/teskeid/weather/travel/route.ts:208`
- `app/api/teskeid/weather/travel/route.ts:239`
- `app/api/teskeid/weather/travel/route.ts:265`

This is not a privacy or RLS issue, but it is a reliability issue for TODO #69's main goal: measuring how often Teskeiðar features are used, especially how often new weather routes are calculated. In a serverless runtime, an unawaited async insert can be cancelled or never complete after the response has been sent. The helper swallows insert errors, which is good for UX, but the current call pattern can silently undercount events.

Recommendation: before using this as release-quality admin telemetry, Claude Code should either:

1. `await recordTeskeidUsageEvent(...)` at the event points, accepting the small extra latency because the helper already catches/logs failures, or
2. use an explicit framework/runtime background mechanism that is guaranteed to run after response, if available and appropriate in this Next/Vercel setup.

Add/adjust tests enough to lock the intended behavior. If the decision is to keep fire-and-forget temporarily, document that the admin numbers are best-effort only.

### Low - distinct route pairs can look broken when `USAGE_EVENT_SECRET` is missing

The v003 privacy finding is fixed: `routePairFingerprint` now returns `null` when `USAGE_EVENT_SECRET` is unset (`lib/teskeid/usage.server.ts:63` to `lib/teskeid/usage.server.ts:68`), and callers omit `routePairHash` from metadata (`app/api/teskeid/weather/travel/routes/route.ts:85` to `app/api/teskeid/weather/travel/routes/route.ts:86`, `app/api/teskeid/weather/travel/route.ts:175` to `app/api/teskeid/weather/travel/route.ts:176`).

That is the right safe default. The remaining product/ops issue is that the admin UI will show route calculations increasing while `Ólík leiðapör` remains `0` if the secret is not configured. That can be misread as a bug in the feature rather than a missing server env var.

Recommendation: either add a small admin note when fingerprinting is disabled, or document clearly in the handoff/release checklist that `USAGE_EVENT_SECRET` must be set before expecting distinct route-pair counts. This is not a blocker if Stebbi is comfortable with the operational caveat.
