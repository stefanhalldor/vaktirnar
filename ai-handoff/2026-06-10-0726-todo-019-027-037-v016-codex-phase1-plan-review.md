## Findings

### 1. High: Phase B app-level pre-read is not safely authorized

Plan reference: v015 lines 91-106.

The plan says a service-role pre-read from `loan_items` is authorized because
`updateLoan` has passed `guardLoanAccess()`. That is not sufficient.

`guardLoanAccess()` proves only:

- the user has a valid Teskeið session,
- `LOANS_ENABLED` is true,
- the user can access the loans feature generally.

It does **not** prove that the user may read or mutate the specific `loanId`.
That per-loan authorization happens inside RPCs such as `update_loan` and
`update_loan_item_details`.

Risk:

- app code could read `item_name`, `note`, `loaned_at` and `due_at` for an
  arbitrary `loanId` before the RPC later returns `not_found`;
- even if the values are not intentionally returned, this breaks the clean
  security boundary and creates future leak risk if logging/error handling
  changes.

Codex recommendation:

- Do not use app-level service-role pre-read for Phase B.
- Prefer an authorized RPC boundary that locks, authorizes, updates, and returns
  safe before/after values.
- If Claude Code wants a narrower alternative, create a service-role-only helper
  RPC that takes `p_actor_id` and `p_loan_id`, applies the same authorization
  rules, and returns only safe fields after authorization.

### 2. High/Medium: Proposed 50 cap does not satisfy "all unread" semantics

Plan reference: v015 lines 27-39 and 135-139.

The product requirement from Stebbi is that `Nýlegt` should show all unread
events, not silently hide older unread rows. A 50-row display cap can be
acceptable only if the rest are explicitly reachable or correctly handled.

Problems in the current plan:

- fetching only 51 rows cannot produce an exact `"og X fleiri"` count;
- current `RecentSection` marks all read by sending only the IDs it received;
- if there are 51+ unread rows, `Allt lesið` would only ack the fetched/visible
  rows unless a new server action is added;
- an overflow label saying "Merktu öll lesin til að sjá eldri" is misleading if
  it does not actually mark all unread rows for the user.

Codex recommendation:

- Either truly fetch all unread rows, with a UI that can handle it;
- or fetch a visible page plus an exact unread count and add a separate
  `ackAllUnreadRecentEventsForUser(userId)` server action that updates all
  unread rows owned by the current user.
- Tests must cover more than the cap and confirm "Allt lesið" clears all unread
  rows, including those not rendered.

### 3. Medium: Current UI is not a scrollable all-unread container

Plan reference: v015 line 37.

The plan says `RecentSection` already renders a scrollable list. Current code
renders a normal vertical list inside a rounded container with `overflow-hidden`
for border radius. It is not a scroll container.

Risk:

- rendering 50 rows inline can dominate `/heim` on mobile;
- the primary home content and feature list can be pushed far down;
- this may feel noisy rather than like a calm inbox.

Codex recommendation:

- Phase A should explicitly define mobile UI behavior before implementation:
  for example a compact preview plus drawer, or a capped-height scroll area with
  clear affordance.
- Do not rely on current `RecentSection` structure as "already scrollable".

### 4. Medium: Note payload privacy is contradictory

Plan reference: v015 lines 69-71.

The plan says "No email, no note content", but then says Phase B will include
old/new note values in the event payload.

Codex recommendation:

- Do not include raw note text in `recent_events.payload` in Phase B.
- Use a semantic diff only:
  - `note_added`,
  - `note_removed`,
  - `note_changed`.
- If Stebbi later explicitly wants note contents in event history, treat that as
  a separate privacy/product decision.

### 5. Medium: Event details need a client display contract, not just payload

Plan reference: v015 lines 141-146 and implementation files in the plan.

The plan adds `changes` to `RecentEventPayload`, but current client display type
only exposes:

- `id`,
- `label`,
- `href`,
- `viewHref`,
- `isDeleted`.

The drawer currently renders only `drawerEvent.label`. That does not satisfy
Stebbi's request that the user can see what changed, such as the previous due
date when a due date was removed.

Codex recommendation:

- Add an explicit `details` or `detailLines` field to the server-rendered
  `RecentEventDisplay`.
- Render those detail lines in the drawer.
- Keep raw payload server-side; pass only safe, localized display strings to the
  client component where practical.

### 6. Low/Medium: "Recipient always sees invitation in Nýlegt" is too strong

Plan reference: v015 line 123.

Phase A prevents invitation events from being hidden by the top-3 limit, but it
does not fix cases where `loan_invitation_received` was never recorded.

Known gap:

- if the recipient had no Teskeið account when the invitation was sent, no
  `loan_invitation_received` event is recorded under current design.

Codex recommendation:

- Phrase this as "recorded invitation events are no longer hidden by the top-3
  limit".
- Keep retroactive event creation for later account creation as a separate
  design question.

## Approval status

### Phase A

Codex approves the direction, but not the implementation details as written.

Claude Code should revise Phase A to define:

1. whether all unread rows are fetched, or whether there is paging/counting;
2. exact behavior of `Allt lesið` when there are more unread rows than rendered;
3. mobile UI behavior for many unread rows;
4. tests for more than the display cap.

No SQL/RLS concern if Phase A only changes read limit/display and keeps queries
scoped by `user_id` and `ack_at IS NULL`.

### Phase B

Codex does **not** approve Phase B as written.

Blockers:

- service-role app-level pre-read before per-loan authorization;
- raw note content in payload;
- no explicit display contract for event details.

Claude Code should return a revised Phase B plan before implementation.

## Suggested revised path

### Revised Phase A

- Choose one:
  - fetch all unread events, or
  - fetch a visible subset plus exact unread count and explicit "show all" /
    "ack all unread" semantics.
- Add a server action for acking all unread rows by current user if not all rows
  are fetched.
- Add UI that works on 360-460 px mobile.
- Add tests for 4 rows, 51+ rows, mark one read, and mark all read.

### Revised Phase B

- Move before/after diff generation behind an authorized SQL/RPC boundary.
- Return only safe before/after fields after the RPC has confirmed actor access.
- Exclude raw note values from event payload in the first pass.
- Add localized detail lines for the drawer.
- Test due date changed, due date removed, item name changed, note changed
  without content, and unauthorized loan ID.
