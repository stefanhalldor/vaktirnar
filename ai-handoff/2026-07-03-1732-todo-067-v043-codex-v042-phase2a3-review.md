### Blocker 1 - Type-check fails and Google Places loader will throw at runtime

Refs:
- `lib/weather/googleMaps.client.ts:5`
- `lib/weather/googleMaps.client.ts:16`
- `lib/weather/googleMaps.client.ts:17`
- `lib/weather/googleMaps.client.ts:27`

`@googlemaps/js-api-loader` v2.1.1 is installed, but the code uses the deprecated `new Loader()` API. In v2, the `Loader` constructor throws at runtime and the TypeScript declarations do not expose `.setOptions()` or `.importLibrary()` on `Loader`.

`npm run type-check` fails:

```text
lib/weather/googleMaps.client.ts(17,12): error TS2339: Property 'setOptions' does not exist on type 'Loader'.
lib/weather/googleMaps.client.ts(27,22): error TS2339: Property 'importLibrary' does not exist on type 'Loader'.
```

Fix before any Phase 2A4/pre-release:

- Import `{ setOptions, importLibrary }` from `@googlemaps/js-api-loader`.
- Call `setOptions({ key, language: 'is', region: 'IS' })` once.
- Return `importLibrary('places')`.
- Re-run `npm run type-check`.

### Blocker 2 - Route weather guesses first geocoding candidate without user map confirmation

Refs:
- `app/api/teskeid/weather/ask/route.ts:56`
- `app/api/teskeid/weather/ask/route.ts:64`
- `app/api/teskeid/weather/ask/route.ts:71`
- `app/api/teskeid/weather/ask/route.ts:74`
- `app/api/teskeid/weather/ask/route.ts:82`
- `app/api/teskeid/weather/ask/route.ts:89`
- `components/weather/PlaceSearch.tsx:14`
- `app/auth-mvp/vedrid/VedridClient.tsx:90`
- `app/auth-mvp/vedrid/VedridClient.tsx:93`

This violates the main product decision from v027/v037: ambiguous or unknown places should be confirmed visually by the user. Current route flow:

1. Extracts origin/destination text.
2. Resolves curated places first.
3. If not curated, calls Google Geocoding server-side.
4. Takes `candidates[0]`.
5. Computes route weather immediately.

That is dangerous for inputs like `Suðurgata`, repeated street names, vague informal names, or places where Google’s top candidate is wrong. The existing `PlaceSearch` only confirms one generic `confirmedPlace` and the route branch ignores `confirmedPlace`, so it cannot confirm both origin and destination.

Fix before Phase 2A4:

- Either make route weather curated-only for now and return a clear “route endpoints need confirmation” state for unknown/non-curated places.
- Or implement explicit route confirmation state with separate origin and destination candidates, two confirmed endpoints, and a visual from/to confirmation before route weather runs.

Do not ship route weather that silently trusts Google’s first geocoding candidate.

### Medium 2 - “Max” route/weather point caps are not strict

Refs:
- `lib/weather/google.server.ts:39`
- `lib/weather/google.server.ts:44`
- `lib/weather/google.server.ts:51`
- `app/api/teskeid/weather/ask/route.ts:103`
- `app/api/teskeid/weather/ask/route.ts:109`
- `app/api/teskeid/weather/ask/route.ts:110`

Both sampling implementations can exceed the stated cap by appending the last point after sampling. Example: if a route has 160 points and max is 80, step 2 samples 80 points and then appends the last point, producing 81. The same pattern can produce 16 weather points despite the “max 15” claim.

This is not catastrophic, but caps should be true caps before billing-facing route weather expands.
