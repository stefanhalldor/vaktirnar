## Root cause of raw i18n keys

The `.replace('{var}', value)` pattern on `tf()` output does not work with next-intl ICU message format. When `tf('key')` is called without the required variables, next-intl returns the raw message key string rather than substituting. All such calls have been replaced with `tf('key', { var: value })`.

## Place labels — Nominatim (no new API required)

New file: `lib/weather/reverseGeocode.client.ts`

Uses Nominatim (OpenStreetMap) reverse geocoding. No new Google API/SKU needed. Free, no key. Results cached in memory per session by coordinate rounded to 2 decimal places (~1 km grid).

Attribution: "© OpenStreetMap contributors" shown in the details panel when a geocoded name is displayed.

**Label resolution priority per selected point:**
1. Origin point → `originName` prop (no network call)
2. Destination point → `destinationName` prop (no network call)
3. Point with separate forecast coordinate → lazy Nominatim fetch for `forecastLat/forecastLon`
4. Point where route and forecast coords are same → no label, no fetch

**Display in `PointDetailsPanel`:**
- When label resolved: `"Spápunktur met.no nálægt {place}"` (e.g. "Spápunktur met.no nálægt Lokufjalli")
- While loading: `"Spápunktur met.no"` (existing key — no flicker from empty state)
- If no name found: `"Spápunktur met.no"` fallback
- Coordinates always shown as secondary audit lines
- `forecastPointExplanation` shown when forecast point differs from road point
- `"© OpenStreetMap contributors"` shown below when a geocoded name is displayed

**Terminology**: No use of "veðurstöð". Route sample point = "Punktur á leið", forecast grid point = "Spápunktur met.no" / "Spápunktur met.no nálægt {place}".
