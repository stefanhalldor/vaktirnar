## Findings

### P1 - Product-table read can silently truncate forecast rows

File: `lib/weather/providers/vedurstofan.server.ts`
Lines: 232-240

`readVedurstofanProductForStations()` does one Supabase select over `vedurstofan_forecasts_latest` for all station IDs:

```ts
const { data, error } = await admin
  .from('vedurstofan_forecasts_latest')
  .select(...)
  .in('station_id', stationIds)
  .order('forecast_time')
```

For 280 stations, this table is not a 280-row table. It is one row per `(station_id, forecast_time)`. With only 4 forecast rows per station that is already 1120 rows, and the real `time=3h` forecast payload is likely more than 4 rows.

Risk: Supabase/PostgREST responses are commonly capped by API row limits unless the query is paginated. If this select returns only the first page, the UI may show partial forecasts or mark many stations as unavailable even though the warmer/projector populated the product table correctly. The route would still return HTTP 200 and the UI would look "mostly working", which makes this a nasty validation bug.

Why tests did not catch it:
- `weather-vedurstofan-station-explorer-api.test.ts` mocks `readVedurstofanProductForStations`.
- Existing projector/warmer tests do not exercise the new product-table reader with >1000 forecast rows.

Recommended fix before moving on:
- Add pagination inside `readVedurstofanProductForStations`.
- Use deterministic ordering, ideally `.order('station_id').order('forecast_time')`.
- Fetch pages with `.range(from, to)` until the returned page is smaller than the page size.
- Add a unit test where the mocked Supabase product-table read returns at least two pages and the final Map contains all rows for all requested stations.

Supabase reference checked by Codex: the JS `.range(from, to)` modifier is zero-based, inclusive, and respects query order, so pagination should always include an explicit order.

### P2 - The API can hide product-table read failures as "all unavailable"

File: `app/api/teskeid/weather/vedurstofan/stations/route.ts`
Lines: 39-42

The route catches any product-reader exception and returns an empty Map. The reader itself also catches errors and returns partial/empty results.

Fail-open is good for user-facing weather, but this validation screen is also an operational tool. If Supabase read fails, Stebbi may see 280 unavailable stations without a clear signal that the DB read failed.

Recommended follow-up:
- Keep fail-open behavior.
- Add a safe server log message or response-level diagnostic that does not expose secrets/user data. Example: include a non-sensitive `dataSourceStatus: "ok" | "error"` in the station explorer API, or expose it only in admin/debug UI.

This is not as urgent as the pagination issue, but it will make validation less confusing.
