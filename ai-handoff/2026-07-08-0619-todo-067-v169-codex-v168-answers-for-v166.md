### B. Independent map-visibility controls

Stebbi asked Codex to clarify this because the phrase "two separate state objects" was unclear.

Clarification:

The separation is between:

1. departure scrubber filters, and
2. map point visibility filters.

It does not mean "one map state for outbound and one map state for return" in this pass, because return is not in scope.

Implementation model:

- Keep the existing scrubber filter state for the departure-time scrubber, e.g. `outboundHiddenStatuses`.
- Add a separate map visibility state for the active/current result, e.g. `mapHiddenStatuses` or `mapOutboundHiddenStatuses`.
- Default for both should be empty set = show everything.
- Scrubber filters only affect the scrubber.
- Map visibility filters only affect the weather points on the map.
- Neither filter changes the deterministic result, selected route, thresholds or weather calculation.

Expected behaviour:

- Toggling `Gott veður` in the scrubber must not hide green points on the map.
- Toggling `Gott veður` in the map controls must not hide green departure times in the scrubber.
- The map controls should use pill-style buttons with counts, e.g.:
  - `Gott veður (49)`
  - `Óþægilegt (3)`
  - `Hættulegt (0)`
  - `Engin gögn (0)` if relevant
- Counts for map controls should reflect points in the active selected departure/current candidate, not the whole multi-day scrubber horizon.
- If the currently manually selected map point becomes hidden by a map filter, clear manual selection and show the default most relevant/worst visible point if possible.
- If all map point statuses are hidden, keep the map usable and show a small empty/filter state with `Sýna allt`.

### C. Coverage text: use concrete forecast end date, not `næstu X daga`

Stebbi prefers concrete user-facing wording such as:

```txt
... fram til föstudagsins 17. júlí.
```

Do not hardcode 9 days. Do not assume met.no always produces exactly 9 usable days for this flow. Derive the end of coverage from the actual generated candidate / forecast horizon available in the result data.

Preferred Icelandic wording:

```txt
Teskeið hefur metið brottfarartíma á klukkutíma fresti fram til föstudagsins 17. júlí.
```

or, if the route points must be explicit:

```txt
Allir veðurpunktar á valdri leið m.v. brottfarartíma á klukkutíma fresti hafa verið sóttir fram til föstudagsins 17. júlí.
```

Use whichever reads better in the combined card, but keep it short enough for mobile.

Weekday genitive forms for Icelandic:

| Day | Genitive form |
|---|---|
| mánudagur | mánudagsins |
| þriðjudagur | þriðjudagsins |
| miðvikudagur | miðvikudagsins |
| fimmtudagur | fimmtudagsins |
| föstudagur | föstudagsins |
| laugardagur | laugardagsins |
| sunnudagur | sunnudagsins |

If implementing localized weekday genitive formatting becomes too much for this pass, use a simpler safe fallback:

```txt
Teskeið hefur metið brottfarartíma á klukkutíma fresti fram til 17. júlí.
```

or:

```txt
Teskeið hefur metið brottfarartíma á klukkutíma fresti á tímabilinu sem spáin nær yfir.
```

Preference order:

1. `fram til föstudagsins 17. júlí` when formatting is reliable.
2. `fram til 17. júlí` if weekday genitive formatting is risky.
3. `á tímabilinu sem spáin nær yfir` only if no clean end date is available.

For English, use a simple equivalent:

```txt
Teskeið has checked hourly departure times until Friday, July 17.
```

or:

```txt
Teskeið has checked hourly departure times until July 17.
```
