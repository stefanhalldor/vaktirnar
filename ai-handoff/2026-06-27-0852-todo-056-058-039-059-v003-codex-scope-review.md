## Findings fyrst

### 1. SQL í v002 má ekki nota `auth.uid()` í service-role RPC

Claude v002 sýnir SQL-snippet sem notar `auth.uid()`. Þetta á ekki við í núverandi loans-mynstri, því appið kallar RPC með service role úr server action og sendir explicit `p_actor_id`.

Rétt mynstur:

- server action sækir notanda með `guardLoanAccess()`
- RPC tekur `p_actor_id uuid`
- RPC staðfestir `p_actor_id` í `auth.users`
- RPC athugar að `p_actor_id` sé raunverulegur aðili að láninu

Ekki byggja #56/#58 á `auth.uid()` nema öllu call-mynstrinu sé breytt, sem er ekki scope hér.

### 2. Lánsdagsetningar eiga að vera `date`, ekki `timestamptz`

Claude v002 samantekt sýndi `p_loaned_at timestamptz` og `p_due_at timestamptz`. Núverandi schema og form nota `date` fyrir `loaned_at` og `due_at`.

Nota skal:

- `p_loaned_at date`
- `p_due_at date`

Annars er óþarfa timezone-áhætta og ósamræmi við `sql/30_loan_items.sql`.

### 3. History má ekki lesa actor út frá `recent_events.user_id`

`recent_events.user_id` er notandinn sem fær eventið, ekki endilega sá sem framkvæmdi aðgerðina. Þegar actor og counterpart fá sama event getur sama `event_key` verið til á tveimur notendum.

Ef #58 notar `recent_events` sem gagnagrunn þarf að:

- de-duplicate-a eftir `event_key`
- bæta actor metadata í payload fyrir ný events, eða sleppa actor-nafni þegar það er ekki öruggt
- ekki giska á actor út frá móttakanda event-færslunnar

Þetta er mikilvægasta history-gildran.

## Uppfært #56 plan

### Heimildir

Accepted loan edit skal leyfa báðum aðilum:

- `created_by = p_actor_id`
- `lender_user_id = p_actor_id`
- `borrower_user_id = p_actor_id`

Þetta á við um samþykkt eða raunveruleg participant-lán. Pending recipient sem hefur ekki orðið að aðila með `claim` á ekki að fá edit-aðgang bara af því netfang passar.

UI þarf að fylgja server-reglunni. `getLoanCardControls` má ekki áfram fela edit fyrir accepted borrower ef server leyfir honum að breyta.

### SQL

Nota nýtt RPC-nafn, eins og Codex v001 lagði til:

`public.update_loan_item_details_and_dates_with_diff`

Fallið þarf að taka:

- `p_actor_id uuid`
- `p_loan_id uuid`
- `p_item_name text`
- `p_note text`
- `p_loaned_at date`
- `p_due_at date`

Fallið þarf að skila:

- `status text`
- `before_item_name text`
- `before_note text`
- `before_loaned_at date`
- `before_due_at date`
- `counterpart_user_id uuid`

Server-side reglur:

- validate-a `p_actor_id` í `auth.users`
- `FOR UPDATE` á loan row
- óviðkomandi fær `not_found` og engin before-gildi
- participant check: `created_by OR lender_user_id OR borrower_user_id`
- `p_due_at IS NULL OR p_due_at >= p_loaned_at`
- uppfæra aðeins `loan_items` editable fields og `updated_at`
- ekki uppfæra invitation snapshots
- grants aðeins til `service_role`
- revoke frá `PUBLIC`, `anon`, `authenticated`

## #58 history plan

### UI

Setja section neðst á `/auth-mvp/lanad-og-skilad/[id]`, undir núverandi detail-card/action svæði.

Vinnuheiti:

`Ferill hlutarins`

Birting:

- action label, t.d. `Breyttur skiladagur`
- timestamp, t.d. `Miðvikudaginn 24. júní kl. 7:40`
- actor, aðeins ef actor er örugglega þekktur
- optional detail lines fyrir breytingar, sama rólega mynstur og `Ólesið`

### Gagnagrunnur

MVP má nota `recent_events` ef Claude tryggir dedupe og actor-varúð. Öruggara langtíma væri sér `loan_history_events` tafla, en það er stærra scope.

Ef `recent_events` er notað:

- búa til service-role RPC, t.d. `get_loan_event_history(p_actor_id uuid, p_loan_id uuid)`
- RPC staðfestir að `p_actor_id` hafi aðgang að láninu
- query-a events fyrir `entity_type = 'loan'` og `entity_id = p_loan_id`
- de-duplicate-a eftir `event_key`
- raða elst fyrst eða nýjast fyrst með skýrri ákvörðun; Codex mælir með elst fyrst fyrir “feril”
- ekki skila raw recipient email eða raw internal payload

Ef actor á að birtast:

- bæta `actorUserId` eða sambærilegu öruggu actor metadata í payload fyrir ný events sem #56 býr til
- sækja display name server-side fyrir actor ef notandi má sjá það
- fyrir eldri events án actor metadata: sýna action án actor, eða neutral texta, ekki giska
