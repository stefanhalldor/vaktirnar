## Findings first

### Medium - v134 must use Valkostur A for precipitation, not add red precipitation now

Claude's Valkostur B proposes adding a red precipitation threshold (`heavyPrecipMmPerHour: 15.0`) and wiring it into `travel.ts`. I do not recommend doing that in v134.

Why:

- Stebbi's concrete request was to set the default precipitation threshold to `5 mm/klst`.
- Current model has only one user-editable precipitation threshold: `cautionPrecipMmPerHour`.
- `ResolvedTravelThresholds` and `TravelThresholdOverrides` do not include a heavy/red precipitation threshold.
- Adding red precipitation now would create a hardcoded safety rule that the UI cannot explain or let the user tune.
- Rain-only red status is a bigger product decision. We only have hourly precipitation, not road surface, drainage, visibility, flooding, snow/ice, or local road warnings.

Current code context:

- `lib/weather/thresholds.ts:27-30` defines `cautionPrecipMmPerHour: 2.0` and unused `heavyPrecipMmPerHour: 3.0`.
- `lib/weather/travel.ts:91-95` only uses `cautionPrecipMmPerHour` and maps precipitation over threshold to `gult`.
- `lib/weather/types.ts:250-261` has no heavy precipitation override/resolved threshold.

Recommendation for v134:

- Use Valkostur A.
- Change `cautionPrecipMmPerHour` to `5.0`.
- Remove `heavyPrecipMmPerHour` if type-check confirms it is unused.
- Update tests around the old 2.0 boundary.
- Add at least one explicit regression test: `4.9` or `5.0` is green, `5.1` is yellow.

If Stebbi later wants red rain, make that a separate phase with UI, copy, tests, and product discussion.

### Medium - Saved places schema needs a stronger key strategy than `unique(user_id, usage_kind, place_id)`

Claude's correction to avoid `usage_kind = 'both'` is good. Use separate `from` and `to` rows.

But this part needs tightening:

> `unique (user_id, usage_kind, place_id)` is stable

Problems:

- Current client `PlaceResult` in `components/weather/PlaceSearch.tsx:8-13` has no `placeId`, only name/address/lat/lon.
- Server fallback can return Google geocoding `placeId`, but the current client type drops it.
- In Postgres, a unique constraint allows multiple `NULL` values, so `place_id` alone will not dedupe fallback/manual rows where `place_id` is null.

Recommendation for v137 saved places:

- Extend `PlaceResult` and route place types to carry optional `placeId`.
- Store a computed `place_key`:
  - `google:${placeId}` when placeId exists
  - otherwise `coord:${round(lat, 5)},${round(lon, 5)}:${normalizedName}` or similar
- Use `unique (user_id, usage_kind, place_key)`.
- Keep `usage_kind` as only `from` or `to`.
- Add owner-only RLS.

This should not be mixed into v134.

## Notes on A1 PlaceSearch fix

When server fallback returns zero results after Google `fetchFields` fails, set:

```ts
setFetchError(false)
setNoResults(true)
```

When server fallback itself fails, set:

```ts
setNoResults(false)
setFetchError(true)
```

This prevents stale state if the user has moved between error/no-result states.
