## Findings

### Medium - Do not ship `SHORTER_DISTANCE` always-on in the first implementation

v196 correctly identifies the next technical experiment: Place ID routing is confirmed end-to-end, but Google still returns only Route 427 for Garðabær -> Þorlákshöfn. Testing `requestedReferenceRoutes: ["SHORTER_DISTANCE"]` is now justified.

Codex would choose option **(b)** from v196, but preferably as an explicit experiment/feature flag rather than only `NODE_ENV !== "production"`:

- safe first pass: enabled on localhost/dev, off in production by default,
- no silent route behavior change for real users,
- easy to promote later if Stebbi validates it on several Iceland routes.

Reason: Google's shorter-distance route feature is explicitly Experimental/pre-GA and prioritizes distance over speed/comfort. Google documents that it can choose local roads, dirt roads, or parking lots when legal. This is useful for diagnosing Þorlákshöfn, but too broad to turn on globally without route-by-route validation.

Official docs checked:

- Shorter distance routes: https://developers.google.com/maps/documentation/routes/shorter-distance-routes
- `computeRoutes`: https://developers.google.com/maps/documentation/routes/reference/rest/v2/TopLevel/computeRoutes
- `Waypoint.placeId`: https://developers.google.com/maps/documentation/routes/reference/rest/v2/Waypoint

### Medium - Route option labeling/ordering needs to be explicit before adding `SHORTER_DISTANCE`

v196 says to "sort all routes by `durationS` ascending as before", but current `getRouteOptions` does not sort. It maps and returns Google's response order.

Relevant code:

- `lib/weather/google.server.ts`: `routeOptions = data.routes.map(...)`, then returned without sorting.
- `app/auth-mvp/vedrid/FerdalagidClient.tsx`: auto-selects `options[0].id`.
- `components/weather/RouteSelectionStep.tsx`: labels `idx === 0` as `routeOptionShortest`, whose Icelandic text is currently `Fljótlegasta leið`.

Before adding `SHORTER_DISTANCE`, decide the intended behavior:

- If options are sorted by duration, `options[0]` is the fastest and auto-selecting it is coherent.
- If Google order is preserved, `idx === 0` means "first returned by Google", not necessarily fastest.
- `SHORTER_DISTANCE` should have its own translated label, e.g. `routeOptionShorterDistance`, instead of relying on index or hardcoded UI text.

Codex recommendation: sort by `durationS` for route picker display and auto-selection, but label by route labels first:

- `SHORTER_DISTANCE` -> `Styttri leið`
- `DEFAULT_ROUTE` -> `Sjálfgefin Google-leið` unless it is also first/fastest
- first fastest route -> `Fljótlegasta leið`
- fallback -> `Önnur leið`

Add labels in both `messages/is.json` and `messages/en.json`.

### Medium - Deduplication should not hide that Google returned `SHORTER_DISTANCE`

v196 says to deduplicate if the `SHORTER_DISTANCE` route has the same fingerprint as a standard alternative. That is right, but the implementation should merge labels or at least log that the shorter-distance route was the same geometry.

Do not simply keep the first route and discard the later label silently. For diagnostics, Stebbi needs to know whether Google:

- returned a distinct shorter route,
- returned the same route with a `SHORTER_DISTANCE` label,
- returned no shorter-distance route.

Also, do not include `routeToken` in route IDs or fingerprints. Google's docs describe `routeToken` as opaque and not stable for comparison across requests. The existing geometry-based ID approach is better.

### Medium - Saved-place `place_id` persistence should be a separate second step, not parallel with the experiment

Proposed B is useful, but it does not fix the confirmed Þorlákshöfn problem by itself. We now know that even with real Place IDs, Google still returns Route 427 unless route strategy changes.

So B should not be implemented in parallel with A unless Stebbi explicitly approves the broader schema/API scope. It touches:

- SQL migration,
- RLS-protected user data,
- saved-place API shape,
- client types and payloads.

Codex recommendation:

1. Run A first as a small provider/UI experiment.
2. If `SHORTER_DISTANCE` finds Route 39 and the route is acceptable, then decide whether to ship it.
3. After route strategy is selected, implement saved-place `place_id` persistence so returning users get the same fidelity from saved/recent selections.

### Low - Keep the field mask requirements precise

v196 is correct that shorter-distance requests need `routes.routeLabels` and `routes.routeToken` in the field mask. The existing app also needs `routes.polyline`, `routes.distanceMeters`, `routes.duration`, and likely `routes.description` for current UI/geometry behavior.

The implementation should keep the current required app fields and add `routes.routeToken`; it should not reduce the field mask to only the docs' minimal example.
