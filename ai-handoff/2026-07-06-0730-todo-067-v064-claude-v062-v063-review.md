## Mat á v062 (auditability)

### Samþykkt

- `WorstMetric` extension með `forecastLat`, `forecastLon`, `metnoUrl` er rétt tæknilega. `forecastLat/forecastLon` = `roundCoord(lat/lon)` — eins og `metno.server.ts` notar (lína 88 þar).
- met.no audit URL er opinbert API, engin leyndarmál lekin út.
- Google Maps link (`https://www.google.com/maps/search/?api=1&query={lat},{lon}`) er opinion. Örugg til birtingar.
- Main card human-friendly, tæknileg gögn á bak við disclosure — rétt UX áhersla.

### Útfærsluákvarðanir sem Claude Code tekur sjálfur

**1. Hvar er `forecastLat/forecastLon` reiknað?**

Í `route.ts` þegar `weatherPoints` eru byggð — þar sem `roundCoord` er til taks (via `import { roundCoord } from '@/lib/weather/places'`). `TravelPointForecast` fær nýja reiti: `forecastLat`, `forecastLon`. `travel.ts` fær þá sem hluta af `TravelPointForecast` og notar þá til að fylla í `WorstMetric.forecastLat/forecastLon/metnoUrl`.

**2. `metnoUrl` í `WorstMetric` — Claude Code reiknar þetta frá `forecastLat/forecastLon`:**

```
https://api.met.no/weatherapi/locationforecast/2.0/compact?lat={forecastLat}&lon={forecastLon}
```

## Mat á v063 (route point transparency)

### Samþykkt

- `routeWeatherPoints` í `TravelPlan` er vel hugsað data model.
- Flags: `isOrigin`, `isDestinationClosest`, `isHighlightedIssue` eru skýrar og einfaldar.
- Beta transparency copy er skynsamleg.
- Ekki reintroduce lodging — einungis transparency.

### Útfærsluákvarðanir sem Claude Code tekur sjálfur

**1. `summaryForWindow` — hvaða gluggi?**

Notað er BEST outbound candidate window (eða single candidate ef windowMode=false). Þetta gefur nothægar upplýsingar. Claude Code útfærir þetta sem auka pass yfir `pointForecasts` eftir að besta candidate er fundinn.

**2. `totalRouteWeatherPoints`**

Notað er `pointForecasts.length` (fjöldi points sem tókst að sækja spá fyrir). Þetta er nothægt — ef einhver points fóru úr, er það synilegt í `routeIndex` gaps. Betri útgáfa (total sampled vs. succeeded) gæti komið seinna.

**3. UI — text list með links, ekki fulla static map með route line**

v063 segir: "Acceptable MVP if full marker map is too much: route summary card with point list." Claude Code mun nota þennan valkost:
- "Spápunktar á leiðinni" expandable section
- Hvert point: `Punktur N af M`, fjarlægð frá uppruna, verstu gildi, links
- Worst point og destination-nearest point merktar

Static map með route line er of stórt scope fyrir þennan pass og krefst providers terms review.

**4. Build order í `checkTravelWeather`**

1. Compute candidates → find best candidate
2. Build `highlightedIssue` (þegar til)
3. Build `routeWeatherPoints` með `isHighlightedIssue` flag á punktinn sem kemur úr `highlightedIssue`
4. Setja `routeWeatherPoints` á `travelPlan`
