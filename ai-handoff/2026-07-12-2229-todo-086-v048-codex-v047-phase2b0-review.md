### 1. Fix before release: hardcoded user-facing strings in the Elta veðrið client

File: `app/auth-mvp/vedrid/elta-vedrid/VedurstofanStationExplorerClient.tsx`

Lines seen in local review:

- `333-338`: table headers are hardcoded Icelandic: `Tími`, `Vindur`, `Átt`, `Úrkoma`, `Hiti`, `Veðurlag`
- `371`: `Parse errors ({station.parseErrors.length})` is hardcoded English/debug text

Why it matters:

- `AGENTS.md` and `Design.md` both require user-facing text to live in `messages/is.json` and `messages/en.json`.
- English UI would still show Icelandic table headers.
- Even though this is validation-first, v046 explicitly framed this as future-ready product infrastructure, not throwaway debug UI.

Recommended fix:

- Add message keys under `teskeid.vedrid.eltaVedrid` in both `messages/is.json` and `messages/en.json`, for example:
  - `forecastTime`
  - `forecastWind`
  - `forecastDirection`
  - `forecastPrecipitation`
  - `forecastTemperature`
  - `forecastWeather`
  - `parseErrors`
- Replace the hardcoded table headers and details summary with `t(...)`.
- Keep units like `m/s`, `°C`, and the empty dash as data formatting unless Claude Code wants to reuse existing forecast unit keys. The release-blocking part is the visible labels.

Severity: release-blocking but low implementation risk.

## What looked good

- The page is correctly guarded by `guardTeskeidSession()` and `guardFeatureAccess(user.email!, 'vedrid')`.
- The API route is gated by `AUTH_MVP_ENABLED`, `WEATHER_ENABLED`, authenticated user, and `vedrid` feature access.
- The API does not expose user id, email, or Supabase internals in the response.
- The Veðurstofan fetch path is fail-open: if the fetch throws, the route still returns station metadata with unavailable status.
