## Step 5 ŌĆö `lib/weather/travel.ts`

### 5a. Add `findNextCaution()` helper

Place before `checkTravelWeather`. It scans future outbound departures starting from `scanFromIso`, stepping hourly, up to 48h or until forecast runs out.

```ts
const NEXT_CAUTION_STEP_S = 3600
const NEXT_CAUTION_MAX_H = 48

function findNextCaution(
  scanFromIso: string,
  durationS: number,
  pointForecasts: TravelPointForecast[],
  trailerKind: 'none' | TrailerKind,
  totalDistanceM: number,
  originName: string,
): NextCaution {
  // Last usable departure = (minimum last forecast hour across all points) - durationS
  let lastForecastMs = Infinity
  for (const pf of pointForecasts) {
    if (pf.hours.length > 0) {
      const last = new Date(pf.hours[pf.hours.length - 1].time).getTime()
      if (last < lastForecastMs) lastForecastMs = last
    }
  }
  const coverageCapMs = isFinite(lastForecastMs) ? lastForecastMs - durationS * 1000 : 0

  const scanStartMs = new Date(scanFromIso).getTime()
  const hardCapMs = scanStartMs + NEXT_CAUTION_MAX_H * 3600 * 1000
  const endMs = Math.min(coverageCapMs, hardCapMs)
  const scannedHours = Math.max(0, Math.round((endMs - scanStartMs) / 3600_000))

  let t = scanStartMs
  while (t <= endMs) {
    const depIso = new Date(t).toISOString()
    const arrIso = addSeconds(depIso, durationS)
    const cand = evaluateCandidate(depIso, arrIso, pointForecasts, trailerKind, totalDistanceM)
    // Skip no_data ŌĆö data absence alone is not a "next caution" we want to highlight
    if (cand.status !== 'graent' && cand.reasonCode !== 'no_data') {
      const issue = buildHighlightedIssue(cand, undefined)
      if (issue && issue.distanceFromOriginM !== undefined) {
        issue.distanceFromLegStartM = Math.round(issue.distanceFromOriginM)
        issue.legStartName = originName
      }
      return {
        departureIso: depIso,
        arrivalIso: arrIso,
        status: cand.status as Exclude<WeatherStatus, 'graent'>,
        reasonCode: cand.reasonCode,
        reasonText: reasonToText(cand.reasonCode),
        issue: issue ?? undefined,
        scannedHours,
      }
    }
    t += NEXT_CAUTION_STEP_S * 1000
  }

  return { scannedHours }
}
```

Note: `buildHighlightedIssue(cand, undefined)` ŌĆö second arg is `returnWorst`. Passing `undefined` means it only considers outbound. This is correct since `nextCaution` is always outbound.

## Open questions for Codex review

1. `nextCaution` scans only outbound from `earliestDeparture + 1h` (or `latestDeparture + 1h` in window mode). Should it instead always scan from the first future hour regardless of window mode? The current logic means window-mode users see "next caution after your window closes", which seems right.

2. `no_data` candidates are skipped in `findNextCaution`. Is this correct, or should lack of forecast data also show as a next caution?

3. `reasonText` is Icelandic regardless of locale (same as `svar`). This is acceptable for MVP since the whole tool is Icelandic-first. Confirm or flag.

4. `scannedHours` is computed as `floor((endMs - scanStartMs) / 3600_000)`. If forecast data runs out at hour 30, this returns `30` not `48`. This is more honest than always saying 48. Confirm this is the right number to show in the "no caution found" message.
