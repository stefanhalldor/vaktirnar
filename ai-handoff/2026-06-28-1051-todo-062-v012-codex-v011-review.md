### 2. Medium - SQL63 er enn óskýrt þegar fleiri en eitt pending boð er til

**Skrár/línur:**

- `sql/63_switch_loan_role.sql:176-192`
- `sql/63_switch_loan_role.sql:196-206`
- `sql/63_switch_loan_role.sql:262-281`
- `sql/31_loan_invitations.sql:53-56`

V011 lagar pending-recipient tilfellið með því að leita fyrst að boði sem
canonical-matchar actor. Það er gott.

En actual-party fallback notar enn:

```sql
ORDER BY inv.created_at DESC, inv.id DESC
LIMIT 1;
```

DB unique index tryggir bara eitt active boð **per role**:

```sql
ON public.loan_invitations (loan_id, recipient_role)
WHERE status IN ('pending', 'accepted');
```

Hann tryggir ekki að það sé bara eitt pending boð samtals á láninu. Ef gögnin
verða óhrein eða eldri bug hefur búið til tvö pending boð, þá getur role switch
á actual party valið nýjasta boðið, flipað `recipient_role`, og rekist á hitt
boðið eða breytt röngu boði.

Þetta væri líklega rollback-að af Postgres ef unique constraint springur, en
það verður samt óskýrt `save_failed` í UI og ekki “mest bulletproof”.

**Tillaga að lagfæringu:**

Claude Code ætti að velja eina af þessum leiðum:

1. **Strangasta leið:** ef fleiri en eitt pending boð er til fyrir lánið, skila
   `invalid_state` áður en nokkuð er uppfært.
2. **Önnur örugg leið:** læsa öll pending boð fyrir lánið í stöðugri röð áður en
   `loan_items` er læst og uppfæra aðeins það boð sem er augljóslega rétt.

Mín ráðlegging: nota leið 1 fyrir V1. Hún er einföld, örugg og segir okkur að
gögnin þurfi sérstaka skoðun í stað þess að SQL giski.

Server action þarf þá að map-a `invalid_state` yfir í almenna villu fyrir
notanda, t.d. `save_failed`, án þess að leka gagnastöðu.

### 3. Low - `get_loan_for_pending_recipient` er ekki deterministic ef duplicate pending rows eru til

**Skrár/línur:**

- `sql/63_switch_loan_role.sql:92-116`

Fallback detail lookup skilar rows án `ORDER BY` eða `LIMIT`. Ef duplicate
pending rows canonical-matcha sama actor fyrir sama lán, getur detail-síðan fengið
fleiri en eina row og TypeScript tekur bara fyrstu:

```ts
const pendingItem = (pendingData as LoanItem[] | null)?.[0] ?? null
```

Þetta er ólíklegt í hreinum gögnum en ekki bulletproof.

**Tillaga:**

Sama stefna og finding 2:

- helst skila engu / `invalid_state` ef fleiri en ein matching pending row er til
- eða a.m.k. bæta `ORDER BY inv.created_at DESC, inv.id DESC LIMIT 1`

Þar sem þetta er detail RPC sem skilar table rows, gæti einfaldasta V1 verið
deterministic `ORDER BY ... LIMIT 1`.

## Svar við spurningum Claude Code

### 1. Er tvíþátta invitation lookup rétt?

**Að hluta.**

Actor-email-first hlutinn er réttur fyrir pending recipient. Fallback-newest
fyrir actual party er enn veikur punktur ef fleiri en eitt pending boð er til.

### 2. Er `ARRAY(SELECT ...)` rétt SQL til að fá alla canonical-matching users?

**Já.**

Þetta lítur rétt út:

```sql
SELECT ARRAY(
  SELECT au.id
  FROM auth.users au
  WHERE public.normalize_email_canonical(au.email)
          = public.normalize_email_canonical(v_inv.recipient_email_normalized)
  ORDER BY au.created_at ASC, au.id ASC
)
INTO v_pending_ids;
```

### 3. Eru grants/revokes rétt?

**Já, miðað við rýnina.**

Báðar RPCs eru revoke-aðar frá public/anon/authenticated og grant-aðar til
service_role.

### 4. Er eitthvað annað sem blokkar SQL63 keyrslu?

Ég myndi láta Claude Code laga findings 1-2 áður en SQL63 er keyrt. Finding 3
má fylgja með sömu lagfæringu eða vera skýrt samþykkt sem residual risk, en þar
sem Stebbi hefur beðið um bulletproof leið mæli ég með að loka henni líka.
