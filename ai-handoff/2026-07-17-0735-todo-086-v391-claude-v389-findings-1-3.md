## Hvað var gert

### Finding 1: Segment proximity í stað vertex proximity

**Rót vandans:** `routePassesNear()` í `routeCautions.ts` og gate matching í `routeControlPoints.ts` notuðu bæði vertex-only proximity:
```ts
return points.some(p => haversineM(p, target) <= radiusM)
```

Ef Google polyline setur ekki vertex nálægt detection point, getur detection dottið niður þótt route segmentið fari í raun í gegnum radius.

**Fix:**

**`lib/weather/providerRouteMatching.ts`** — nýtt exported helper bætt við:
```ts
export function pointToPolylineDistanceM(
  lat: number, lon: number,
  polyline: ReadonlyArray<{ lat: number; lon: number }>,
): number
```
Notar clamped projection á hvert segment (flat-earth, gild fyrir ≤500 km). Skilar Infinity fyrir tóma polyline. Sama stærðfræði og er í `projectToPolyline` / `matchProviderPointsToRoute` — sama kóðagrundvöllur.

**`lib/weather/routeCautions.ts`:**
- Importar `pointToPolylineDistanceM` frá `./providerRouteMatching`
- Fjarlægt local `haversineM` (var aðeins notað í `routePassesNear`)
- `routePassesNear` uppfært til að nota segment projection:
  ```ts
  function routePassesNear(points, target, radiusM): boolean {
    return pointToPolylineDistanceM(target.lat, target.lon, points) <= radiusM
  }
  ```

**`lib/weather/routeControlPoints.ts`:**
- Importar `pointToPolylineDistanceM`
- Gate matching uppfært:
  ```ts
  const allGatesMatched = section.gates.every(gate =>
    pointToPolylineDistanceM(gate.lat, gate.lon, result) <= gate.radiusM
  )
  ```
  Injection boundary index (startIdx/endIdx) notar enn vertex proximity — það er rétt, við þurfum index.

### Finding 3: Reversed route injectar anchors í ranga röð

**Rót vandans:** Þegar `startIdx > endIdx` var swap-ið en anchors voru alltaf sett inn í registry order (west→east), þótt leiðin fari east→west.

**Fix** í `augmentProviderMatchingPoints`:
```ts
let reversed = false
if (startIdx > endIdx) {
  ;[startIdx, endIdx] = [endIdx, startIdx]
  reversed = true
}
const orderedAnchors = reversed ? [...section.anchors].reverse() : [...section.anchors]
result = [
  ...result.slice(0, startIdx + 1),
  ...orderedAnchors,
  ...result.slice(endIdx),
]
```

Anchors eru nú alltaf í ferðastefnu röð, sem tryggir rétt `distanceFromOriginM` og `routeFraction` fyrir Veðurstofustöðvar.

## Prófaðar öryggisáhrif á gildandi próf

Segment proximity er ALLTAF ≤ vertex proximity. Því:
- Öll próf sem stóðust áður (vertex innan radius) standa enn (segment er jafn eða nær).
- Einu nýja hegðunin er að segment-only tilfelli sem dóttu áður virka nú.

Stóra prófið sem breytist í hegðun: reversed route anchor order — fyrri hegðun var registry order (west→east) óháð ferðastefnu. Nú er hún travel-order.
