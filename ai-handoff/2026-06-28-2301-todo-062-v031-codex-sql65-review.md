## Finding

### Medium - rollback leiðbeiningin í SQL65 myndi drop-a pending fallback fallið

**Skrá:** `sql/65_fix_switch_loan_role_security_definer.sql`

Í rollback-kaflanum stendur:

```sql
DROP FUNCTION IF EXISTS public.switch_loan_role(uuid, uuid);
DROP FUNCTION IF EXISTS public.get_loan_for_pending_recipient(uuid, uuid);
Then re-run SQL64 without SECURITY DEFINER.
```

Þetta er ekki öruggt sem rollback-lýsing, því SQL64 endurskapar bara
`switch_loan_role`. SQL64 endurskapar ekki `get_loan_for_pending_recipient`.

Afleiðing: ef einhver fylgir rollback-textanum myndi `get_loan_for_pending_recipient`
hverfa og pending-recipient edit/detail fallback gæti brotnað.

**Lagfæring:** Breyta rollback-kaflanum þannig að hann segi annaðhvort:

- endurkeyra SQL63 og síðan SQL64, ef ætlunin er að fara aftur í pre-SQL65
  `SECURITY INVOKER` stöðu; eða
- endurskapa bæði föllin án `SECURITY DEFINER` með fullri SQL skilgreiningu.

Ég mæli með fyrra orðalaginu í commentinu:

```txt
Rollback (reverts to SECURITY INVOKER and re-introduces this bug):
  1. Re-run sql/63_switch_loan_role.sql to restore both functions.
  2. Re-run sql/64_fix_switch_loan_role_ambiguous_status.sql to restore the
     ambiguous-status fix on switch_loan_role.
  3. Reload PostgREST schema cache.
```

Athugið að þetta er rollback-skjalaleiðrétting, ekki business logic breyting.

## Það sem lítur rétt út í SQL65

- `switch_loan_role` fær `SECURITY DEFINER`.
- `get_loan_for_pending_recipient` fær `SECURITY DEFINER`.
- Bæði föll halda `SET search_path = ''`.
- Bæði föll halda `EXECUTE` grant eingöngu á `service_role`.
- `PUBLIC`, `anon` og `authenticated` eru áfram revoke-uð.
- Enginn beinn table grant á `auth.users` er bætt við.
- Engin user-facing gögn eða netföng eru sett í return contract.
- Þetta passar við staðfestu villuna:
  - direct SQL sem `postgres` virkar
  - PostgREST RPC sem app/service-role fær `42501 permission denied for table users`
