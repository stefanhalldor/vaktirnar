## Findings

### P1 - Return heatmap and map panel use outbound distance as if it were distance from destination

The return leg is now visually separated, which is the right product direction. But the distance data is still wrong for return selections.

`components/weather/travelAuditMap.helpers.ts` builds a `TravelIssue` from a selected candidate:

- line 131: `candidateToIssue(c, leg)`
- lines 143 and 161: `distanceFromLegStartM: m?.distanceFromOriginM`

That is correct for outbound. It is not correct for return. For return, `distanceFromOriginM` is still the point's distance from the original origin along the outbound route. The distance from the return start/destination should be:

```ts
routeDistanceM - distanceFromOriginM
```

The same issue exists in the return heatmap slot detail. `components/weather/DepartureHeatmap.tsx` computes:

- line 179: `distKm = Math.round(m.distanceFromOriginM / 1000)`
- line 197: displays `{distKm} km frá {originName}`

For return heatmap, `originName` is set to `destination.name` in `app/auth-mvp/vedrid/FerdalagidClient.tsx` line 503. That means a point 84 km from Garðabær on the outbound route can be displayed as `84 km frá [destination]` on heimferð, even though it may actually be close to the home side or vice versa.

User-visible impact:

- Heimferð details can point to the right weather metric but wrong distance context.
- The user may think the risky point is near the destination/return start when it is actually closer to home, or the opposite.

Fix direction:

- Pass route distance into `candidateToIssue()` for return, or add `distanceFromLegStartM` to candidates/issues server-side.
- For `leg === 'return'`, compute `distanceFromLegStartM = routeDistanceM - distanceFromOriginM`.
- Give `candidateToIssue()` a `legStartName`, so `TravelAuditMap` does not fall back to outbound origin.
- Update `DepartureHeatmap` to support `leg`, `routeDistanceKm/M`, and correct distance display for return.
- Add tests for a return candidate where `distanceFromOriginM` and `routeDistanceM - distanceFromOriginM` are deliberately different.

### P1 - Selecting a green slot can leave the map on the previously selected red/yellow slot while saying it shows the default worst point

V092 intentionally does not move the map for green slots because `candidateToIssue()` returns `undefined` for green candidates:

- `components/weather/travelAuditMap.helpers.ts` line 132: `if (c.status === 'graent') return undefined`
- `app/auth-mvp/vedrid/FerdalagidClient.tsx` lines 163-169: selected green slot makes `heatmapHighlightedIssue` become `undefined`
- `components/weather/TravelAuditMap.tsx` lines 213-220: if there is no `highlightedIssue`, the sync effect returns without changing `selectedIndex`

Then `FerdalagidClient` shows:

```text
Kortið sýnir verstu veðrastöðina á leiðinni. Valin brottfarartíð er með gott veður.
```

But the map selection is not reset to the default/worst issue. If the user first taps a red/yellow slot and then taps a green slot, the selected map point can remain the previous red/yellow slot. The note says "worst point", but the panel may actually be showing the previous selected slot's point with default summary values.

Fix direction:

- Track whether a green slot is active.
- When a green slot is selected, explicitly reset `TravelAuditMap` to the original `result.travelPlan.highlightedIssue`, or a clearly chosen default point.
- Do not pass `undefined` and rely on stale selected state.
- Consider changing the copy to avoid "veðrastöð" unless these are true stations. These are forecast/route points.

### P2 - Date/time formatting hardcodes Icelandic `kl.` in English UI

`components/weather/DepartureHeatmap.tsx` line 20 hardcodes:

```ts
return `${formatDayLabel(isoString, locale)} kl. ${formatKlTime(isoString)}`
```

That affects `SlotDetail` and aria labels in English. The English UI will show labels like `Mon, Jul 7 kl. 18:00`.

Fix direction:

- Put the date/time template into `messages/is.json` and `messages/en.json`, e.g. `{date} kl. {time}` vs `{date} at {time}`.
- Or use an existing localized date/time formatter helper if the project has one.
