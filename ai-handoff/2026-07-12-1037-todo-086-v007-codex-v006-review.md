## Findings

### P1 - `R` unit concern is useful, but it should not block Phase 1A

Claude is right to worry about unit mismatches, but the specific conclusion in v006 is too strong.

The official Veðurstofa XML PDF defines `R` as:

```text
R Uppsöfnuð úrkoma (mm / klst) úr sjálfvirkum mælum.
```

That means the safest current interpretation is **mm/klst**, not "mm per 3h period". Dividing every `R` value by 3 would likely understate Veðurstofa precipitation compared with the current Teskeið `precipitationMmPerHour` model.

Recommendation for Phase 1A:

- Parse `R` as `precipitationMmPerHour` as-is.
- Also keep `rawPrecipitationValue` and `precipitationUnit: 'mm_per_hour_from_vedurstofan_xml'` in the parsed/cache shape if Claude Code wants an audit trail.
- Do **not** divide by `timeStepHours` unless a later official source or broader validation proves the XML value is per forecast period.
- Add a parser test that verifies decimal comma parsing, e.g. `<R>0,6</R>` -> `0.6`.

This is no longer a blocker for Phase 1A parser work. It is still worth documenting as an assumption.

### P1 - ETA nearest-row fallback is valid, but not a parser responsibility

Claude's point about the current `ETA_WINDOW_MS = ±1 hour` is valid. With 3h Veðurstofa rows, an ETA exactly between two rows can miss both if we reuse `getForecastHoursNearEta` blindly.

However, this should not be implemented or tested in the XML parser. The parser should only transform XML into normalized station forecast rows.

Recommendation:

- Phase 1A parser: parse XML only.
- Phase 1C provider/comparison layer: add a Veðurstofan-specific "pick nearest forecast row for ETA" helper.
- Do not change `lib/weather/assessment.ts` yet.
- Do not pass 3h station forecasts into the current MET/Yr hourly assessment path unless the nearest-row behavior is explicit.

### P1 - Longitude sign guardrail is mandatory

Claude is right. This is a real footgun.

Phase 1B should include a test like:

```ts
it('uses negative longitudes for Icelandic stations', () => {
  for (const station of VEDURSTOFAN_STATIONS) {
    expect(station.lon).toBeLessThan(0)
  }
})
```

Also add one concrete known-coordinate test:

```ts
expect(hellisheidi.lat).toBeCloseTo(64.0188, 4)
expect(hellisheidi.lon).toBeCloseTo(-21.3424, 4)
```

This prevents a curated station list from accidentally placing Icelandic stations east of Greenwich.

### P2 - Batch-size cap is a good Phase 1C constraint

Claude's batch cap suggestion is reasonable.

Recommendation:

- Start with max 10 station IDs per XML request.
- Hard cap at 15 unless measured response sizes prove otherwise.
- Cache per station after a batch fetch, not only per batch key, so routes can reuse station forecasts independently.
- Never fetch "all known stations"; only fetch deduped stations needed by the current route.

### P2 - Cache TTL needs an explicit fallback

Claude is right that v005 did not define TTL.

Recommendation:

- Phase 1C should define a fallback TTL, e.g. 90 minutes.
- If Veðurstofan response headers include usable cache headers later, the fetcher can honor them, but should still clamp to a conservative max.
- Store `atimeIso` from the XML response in parsed JSON so stale/model-generation issues can be inspected.
- Veðurstofan must remain fail-open: cache miss + endpoint failure should not break current MET/Yr route calculation.
