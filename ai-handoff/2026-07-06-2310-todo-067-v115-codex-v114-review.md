## Findings

### P1 - Threshold validation still allows nonsensical wind ordering

`app/api/teskeid/weather/travel/route.ts:25-45` validates numeric ranges, but it does not enforce the key invariant from the plan: `cautionWindMs < redWindMs`.

The client has the same gap in `app/auth-mvp/vedrid/FerdalagidClient.tsx:154-180`.

Why this matters:

- Users can set `cautionWindMs = 25` and `redWindMs = 10`.
- The model will mark wind >= 10 as red before it ever reaches caution.
- The UI then presents thresholds that look valid but produce confusing, non-monotonic behavior.

Required fix:

- Validate the resolved values after merging partial overrides with defaults.
- Reject when `cautionWindMs >= redWindMs`.
- Do this both client-side for immediate UX and server-side for safety.
- Add tests for:
  - direct `resolveThresholds`/validation behavior if validation is extracted
  - API route invalid payload if route tests are practical
  - client manual check if component route tests are out of scope

### P2 - Switching time mode does not clear the previous datetime value

v114 says switching mode clears the previous constraint, but current UI only changes `timeConstraintKind`.

Relevant code:

- `FerdalagidClient.tsx:208-212` defines the three options.
- `FerdalagidClient.tsx:331-343` calls `setTimeConstraintKind(opt.value)`.
- `timeConstraintValue` is not cleared.

This means a user can enter a departure time, tap `Heim fyrir`, and the same visible datetime is now interpreted as a home deadline. It is not hidden, but it still carries a value across a semantic mode change in a way that can surprise people.

Recommended fix:

- On mode change, clear `timeConstraintValue`.
- Alternatively keep it only if Stebbi explicitly wants "reuse same date/time across modes", but then the copy should make that behavior obvious.
- The current v113/v114 direction was to avoid stale time constraints, so clearing is the safer default.

### P2 - API still accepts multiple time constraints

The client now sends only one of `earliestDepartureAt`, `latestArrivalBy`, `latestHomeBy`, but `app/api/teskeid/weather/travel/route.ts:91-103` still accepts all three if a caller sends them.

This is not a public unauthenticated API, but it is still the server boundary for the feature. The deterministic model is now conceptually "one active time question", so the server should enforce that contract.

Recommended fix:

- Count the provided time fields after validation.
- If more than one is present, return `400` with a clear error such as `time_constraint_conflict`.
- Add this to `errMap` in `FerdalagidClient`.
- Add at least one test if route tests are already available; otherwise document as manual/API QA.

### P2 - Threshold defaults are duplicated in client and server

`FerdalagidClient.tsx:32-38` hardcodes:

```ts
const DRIVING_DEFAULTS = { cautionWindMs: 15, redWindMs: 20, redGustMs: 28, cautionPrecipMmPerHour: 2.0 }
const TRAILER_DEFAULTS = { cautionWindMs: 13, redWindMs: 18, redGustMs: 25, cautionPrecipMmPerHour: 2.0 }
```

The server source is `lib/weather/thresholds.ts`.

This duplication can drift later and is risky because the whole point of this feature is "we show the exact thresholds used". Prefer a shared exported helper/default object from `lib/weather/thresholds.ts` if it is safe to import into the client bundle.

Recommended fix:

- Export a client-safe `defaultTravelThresholdsForTrailerKind(...)` or reuse `resolveThresholds(...)` on the client.
- Keep one source of truth for default threshold numbers.

### P2 - Threshold step action row is likely cramped on 360px mobile

`FerdalagidClient.tsx:424-448` renders three actions in one row:

- back
- `Nota sjálfgefin viðmið`
- `Reikna ferðina`

That is a lot of text for a 360px viewport. It may technically fit by squeezing, but it is likely to feel cramped or cause awkward wrapping/overflow.

Recommended fix:

- Put `Nota sjálfgefin viðmið` on its own quieter row/link.
- Keep bottom action row to back + primary submit.
- Confirm at 360px, 390px and 460px.

### P3 - API validation tests are still missing

v114 added good model-level threshold tests in `lib/__tests__/weather-travel.test.ts:781-853`, and both `npm run type-check` and `npm run test:run` pass.

However, the risky part of threshold validation is in the API route. No test currently proves invalid API payloads return `400 thresholds_invalid`, and no test covers multi-time-conflict because that validation does not exist yet.

This can be acceptable for the immediate local loop, but before production confidence I would add route-level tests or extract validation functions into testable helpers.
