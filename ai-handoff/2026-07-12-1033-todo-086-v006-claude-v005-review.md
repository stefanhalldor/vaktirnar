## P1 — `R` einingar óstaðfestar

Codex nefnir ekki hvort `R` (úrkoma) úr Veðurstofunni sé **mm per tímabil** eða **mm/klst**.

Þetta skiptir máli: MET `precipitation_amount` er:
- `next_1_hours` → mm í næstu klukkustund = mm/klst
- `next_6_hours` → mm í 6 klst → `forecast.ts` deilir með 6 til að fá mm/klst

Veðurstofan `R` er með 3h spástigi. Ef `R=0.6` þýðir það 0.6 mm á 3 klst → 0.2 mm/klst, ekki 0.6 mm/klst.

**Áhrifin:** ef við notum `R` beint sem `precipitationMmPerHour` án deilingar, þreföldum við úrkomunýstigið miðað við MET. Þetta myndi gera Veðurstofuna að líta út eins og hún sé alltaf þrefalt meira rignandi.

**Þarf að staðfesta:** les PDF docs eða proves með þekktri stöð og þekktu tímabili. Bæta við divisor (`/ timeStepHours`) í parser þegar `R` er normalized í `precipitationMmPerHour`.

**Tillaga í cache shape:** geyma raw `R` value og `timeStepHours` sérstaklega svo við getum normalized seinna án re-fetch.

## P1 — ETA window passar ekki við 3h spástigi

`assessment.ts` notar `ETA_WINDOW_MS = ±1 klst` (1 hour window):

```ts
const ETA_WINDOW_MS = 3_600_000 // ±1 hour around each route point's estimated arrival time
```

MET er hourly, þannig ±1 klst fangar alltaf að minnsta kosti eitt row.

Veðurstofan er 3h-stepping. Ef ETA er t.d. 12:30 og spágildi eru 12:00 og 15:00:
- 12:00 → 30 mín frá ETA → **innan** ±1 klst ✓
- En ef ETA er 13:30: 12:00 er 1:30 frá ETA, 15:00 er 1:30 frá ETA → **báðar utan** ±1 klst window → **engin row finnst**

Í versta tilfelli (ETA nákvæmlega á milli 3h rows) gæti `getForecastHoursNearEta` skilað tómu fylki fyrir Veðurstofu-gögn.

**Lausn:** Í Veðurstofu-provider, nota **nearest-row** logic frekar en window-filter þegar sami domain seam er notaður:

```ts
// Þegar window skilar [], fara í nearest row sem fallback:
function nearestForecastRow(hours: HourPoint[], etaMs: number): HourPoint | null {
  if (!hours.length) return null
  return hours.reduce((best, h) =>
    Math.abs(new Date(h.time).getTime() - etaMs) <
    Math.abs(new Date(best.time).getTime() - etaMs) ? h : best
  )
}
```

Þetta á að vera í Veðurstofu provider/comparison layer, **ekki í `assessment.ts`** (breyting á domain seam er ekki hluti af Phase 1).

## P1 — Lon sign conversion er P0 bug risk

Codex nefnir koordinátusnúning:

> "Veðurstofan station pages display Icelandic west longitudes as positive-looking values"

Þetta er tæknilega rétt. En það er ekki bara "display format" vandinn — ef við byggjum curated station list í Phase 1B og gleymir að negere longitude þá verða öll station coordinates rangar og haversine distance útreikningar munu gefa villandi station mapping.

Dæmi:
- Hellisheiði: `(64.0188, 21.3424)` á station page → rétt WGS84: `lat=64.0188, lon=-21.3424`
- Ef við gleymir minus: `lon=21.3424` → þetta er í Kazakhstanar!

**Guardrail:** curated station list (Phase 1B) ætti að hafa TypeScript test sem staðfestir að allar lon-values eru neikvæðar (eða 0 ef stöðin er á `lon=0`, sem er ólíklegt á Íslandi):

```ts
it('all station longitudes are negative (Iceland is west of Greenwich)', () => {
  for (const station of VEDURSTOFAN_STATIONS) {
    expect(station.lon).toBeLessThan(0)
  }
})
```

## P2 — Batch size limit óskýr

Codex mælir með "split into small stable batches if needed" en gefur ekki upp batch-size.

URL-length consideration: `ids=31392;571;5544;...` með 30 stöðvum gæti orðið ~200+ tákn. Þetta er vel undir venjulega URL length limits (8192 bytes), þannig að batch-size-of-all-stations-per-route er líklega í lagi tæknilega.

Hins vegar: **þjónustustigi** vandamálið er önnur saga. Codex staðfesti 62 rows per station. 30 stöðvar × 62 rows × XML = nokkrar MB af XML per request. Þetta þarf að mæla.

**Tillaga:** byrja á ≤10 stations per batch í Phase 1C sem initial cap, mæla response sizes og auka ef Veðurstofan bregst vel við. Hard cap at 15 stations per request.

## P2 — Cache TTL óskýr fyrir Veðurstofan

MET notar `Expires` header → `parseExpires()` í `metno.server.ts`.

Veðurstofan XML: Codex nefnir ekki hvort `Expires` eða `Last-Modified` header kemur í svari. Mjög líklegt er að þeir séu fjarverandi eða óáreiðanlegir.

Veðurstofan `forec` uppfærist sjaldnar en MET. Með 3h spástigi myndi 90 mínútna TTL vera hóflegt: rétt lægra en eitt spástímabil. Þetta tryggir að við sækjum aldrei eldra en 1.5× spástímabilið.

Codex nefnir ekki þetta. Phase 1C þarf explicit default TTL constant (t.d. `90 * 60 * 1000` ms) sem fallback þegar engin `Expires` header kemur.
