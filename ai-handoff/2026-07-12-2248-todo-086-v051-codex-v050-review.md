### P1 - `elta-vedrid` should be a gated lab layer inside the future `vedrid` flow

Stebbi clarified after v050:

> Í framtíðinni mun elta-vedrid og vedrid vera eitt flæði ... ágætt að þróa það undir nýju elta-vedrid sem er feature-gated?

Codex recommendation: yes.

Develop it now under a separate `elta-vedrid` feature gate, but architect it as a temporary validation/lab layer for the future `vedrid` flow, not as a permanently separate Teskeið.

Practical implication:

- `vedrid` remains the baseline product/weather access.
- `elta-vedrid` is the extra per-user gate for people allowed to see the experimental station validation layer.
- Route can stay nested under `/auth-mvp/vedrid/elta-vedrid`.
- UI copy should frame it as validation/source-inspection, not a separate launched product.
- Later, once validated, this can be folded into normal `vedrid` and the extra gate can be removed.

Implementation decision Claude Code should confirm with Stebbi:

- Prefer requiring both baseline weather access and lab access:
  - `checkFeatureAccess(..., 'vedrid')`
  - `checkFeatureAccess(..., 'elta-vedrid')`
- Or make `elta-vedrid` imply weather access.

Because the page lives under `/auth-mvp/vedrid/...` and links back to `/auth-mvp/vedrid`, Codex prefers the first option: require both. That avoids a user who can open the lab page but cannot open the parent weather page.
