## Findings

### High: History fallback getur tapað stöðvum úr nýjustu Vegagerðar-sótt

Í `lib/weather/providers/vegagerdinCurrent.server.ts` notar `readVegagerdinCurrentFromHistory()` nýjasta `measured_at` sem batch-anchor og les svo bara rows með:

```ts
.gte('measured_at', batchCutoffIso)
```

Relevant lines:

- `lib/weather/providers/vegagerdinCurrent.server.ts:499`
- `lib/weather/providers/vegagerdinCurrent.server.ts:511`
- `lib/weather/providers/vegagerdinCurrent.server.ts:514`
- `lib/weather/providers/vegagerdinCurrent.server.ts:517`

Þetta endurskapar ekki endilega nýjustu upstream-sóttina. Vegagerðin payload getur innihaldið stöðvar sem voru sóttar á sama tíma en með eldri `measured_at`, t.d. ef stöð hefur ekki sent nýja mælingu síðustu 15-40 mínútur. Þá var stöðin samt í nýjasta payloadinu, en history fallback myndi henda henni út ef hún lendir utan 10 mín `measured_at` gluggans.

Þetta er sérstaklega slæmt fyrir markmið Stebba: "það ættu alltaf að vera gögn". Fallback á að varðveita síðustu gagnlegu myndina sem við fengum frá Vegagerðinni, ekki þrengja hana niður í nýjustu mælistöðvarnar.

Recommended fix:

- Nota `last_fetched_at` sem batch-anchor, ekki `measured_at`.
- Finna nýjasta `last_fetched_at` innan 24h.
- Lesa rows sem tilheyra þeirri síðustu fetch-batch, t.d. `last_fetched_at >= newestLastFetchedAt - smallWindow` og `last_fetched_at <= newestLastFetchedAt`.
- Bæta index í SQL83 ef þarf:

```sql
CREATE INDEX IF NOT EXISTS vegagerdin_measurements_history_last_fetched_at_idx
  ON public.vegagerdin_measurements_history (last_fetched_at DESC);
```

- Select-a `last_fetched_at` í provider fallbackinu.
- Láta payload-level `fetchedAtIso` byggja á batch fetch-tímanum, ekki elsta original `fetched_at`.

Ath: `fetched_at` í SQL83 er sett við insert, en `last_fetched_at` uppfærist við conflict. Ef sama mæling birtist í mörgum fetchum má `fetched_at` verða gamalt þótt row hafi verið staðfest nýlega. Þess vegna er `last_fetched_at` réttara fyrir fallback batch.

v507 tengir history fallback við:

- `app/api/teskeid/weather/vegagerdin/current/route.ts`

En eftirfarandi user-facing flæði nota enn `readVegagerdinCurrentFromCache()` beint:

- `app/auth-mvp/vedrid/puls/vegagerdin/stod/[stationId]/page.tsx:4`
- `app/auth-mvp/vedrid/puls/vegagerdin/stod/[stationId]/page.tsx:38`
- `app/api/teskeid/weather/vedurpuls/vegagerdin/stations/[stationId]/preview/route.ts:3`
- `app/api/teskeid/weather/vedurpuls/vegagerdin/stations/[stationId]/preview/route.ts:24`
- `lib/chat/adapters/weather.server.ts:3`
- `lib/chat/adapters/weather.server.ts:50`

Þetta skýrir líklega hluta af því sem Stebbi hefur séð í browser: `/vedrid` getur haft Vegagerðargögn, en púls/detail/thread opnun getur samt fallið á cache-only lookup.

Recommended fix:

- Búa til einn shared helper, t.d. `findVegagerdinCurrentMeasurementByStationId(stationId)`, sem notar `readVegagerdinCurrentWithHistoryFallback()`.
- Nota helperinn í:
  - Vegagerðin pulse page
  - Vegagerðin preview route
  - `buildWeatherPulseTarget(..., provider='vegagerdin')`
- Halda cron anti-stampede og verify áfram cache-only. Þar er cache-only rétt, því cron á að sannreyna cache-write, ekki fela failure á bak við history.

### Medium: SQL83 vantar static migration test coverage

Ég fann ekki `vegagerdin_measurements_history` coverage í `lib/__tests__/sql-migration.test.ts`. Fyrir nýja history töflu með service-role-only RLS ætti að bæta static tests áður en við merkjum þetta lokað.

Testa sérstaklega:

- `sql/83_vegagerdin_measurements_history.sql` exists
- creates `public.vegagerdin_measurements_history`
- primary key `(station_id, measured_at)`
- `first_fetched_at DEFAULT now()`
- `last_fetched_at`
- RLS enabled
- revokes `PUBLIC, anon, authenticated`
- grants only `service_role`
- indexes for `measured_at DESC`, `station_id, measured_at DESC`, and `last_fetched_at DESC` if added
- updated_at trigger
- rollback comment

### Low: History fallback query itself is not directly unit-tested

`lib/__tests__/vegagerdin-history.test.ts` tests `buildPayloadFromHistoryRows()` well, and API test covers pass-through of `cacheStatus: 'history_fallback'`.

What is missing:

- cache unavailable -> history newest batch exists -> fallback returns `history_fallback`
- cache unavailable -> history too old -> unavailable
- history rows with same `last_fetched_at` but mixed `measured_at` are all preserved
- `last_fetched_at` is used for payload `fetchedAtIso`

This does not need a huge integration test. A small mocked Supabase-chain test is enough.
