## Findings

### P1 - Cron auth does not fail closed when `CRON_SECRET` is missing

File: `app/api/cron/warm-vedurstofan/route.ts`
Lines: 9-11

Current code:

```ts
const authHeader = request.headers.get('authorization')
if (authHeader !== `Bearer ${process.env.CRON_SECRET}`) {
  return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
}
```

If `CRON_SECRET` is missing, this comparison becomes `Bearer undefined`. If it is empty, it becomes `Bearer `. That means a bad environment configuration can still be manually invoked by anyone who guesses the bad derived header. This route writes weather data through the service-role warmer/projector path, so it should fail closed before it can do any work.

Vercel's own cron docs recommend this shape:

```ts
const cronSecret = process.env.CRON_SECRET
if (!cronSecret || authHeader !== `Bearer ${cronSecret}`) {
  return new Response('Unauthorized', { status: 401 })
}
```

Required fix before commit/push:

- Read `const cronSecret = process.env.CRON_SECRET`.
- Return 401 if `!cronSecret`.
- Compare against `Bearer ${cronSecret}` only after confirming it exists.
- Add tests:
  - missing `CRON_SECRET` returns 401 and does not call warmer
  - empty `CRON_SECRET` returns 401 and does not call warmer
  - `Authorization: Bearer undefined` does not pass when env is missing

### P2 - Cron delivery can duplicate or miss runs; current job is mostly safe but not locked

Vercel docs say cron delivery is best effort and can occasionally invoke the same scheduled run more than once. They also recommend locks/idempotent reconciliation for reliable cron jobs.

For this job, the risk is acceptable for an initial version because:

- interval is every 6 hours, while `maxDuration` is 300 seconds
- warmer/projector are mostly idempotent: cache writes/upserts and product-table upserts
- duplicate runs are inconvenient and extra load, not directly a user-data leak

Still, this should stay on the TODO 086 residual-risk list. If logs show duplicate work or timeouts, the next hardening should be a DB/run lock or generation marker as part of the exact replace semantics follow-up.
