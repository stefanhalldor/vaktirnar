## Findings

### High - v134 is still too two-provider-specific for a system that must add Vegagerðin next

References:
- v134 section 1: `activeProviderCount = Number(showMetno) + Number(showVedurstofan)`
- v134 section 5: `activeVedurstofanStationPoints`
- v134 `TravelAuditMap` prop: `vedurstofanStationPoints`

The plan fixes the immediate Veðurstofan problems, but it still bakes provider logic into `showMetno` + `showVedurstofan` booleans and a Veðurstofan-specific map prop. Since Vegagerðin is likely next, this risks making the next provider another one-off layer.

Recommended adjustment before implementation:

- Keep the current UI toggles if that is fastest, but introduce a provider-key model underneath:

```ts
type WeatherProviderKey = 'metno' | 'vedurstofan' | 'vegagerdin'

const selectedProviders: Record<WeatherProviderKey, boolean> = {
  metno: showMetno,
  vedurstofan: showVedurstofan,
  vegagerdin: false, // disabled for now
}

const activeProviderKeys = (Object.keys(selectedProviders) as WeatherProviderKey[])
  .filter(key => selectedProviders[key])
```

- Rename map data from `vedurstofanStationPoints` to something generic, for example:

```ts
type ProviderMapPoint = {
  provider: WeatherProviderKey
  lat: number
  lon: number
  id: string
  label: string
  status: WindDisplayStatus
  windMs: number | null
  forecastTimeIso: string | null
  etaIso: string | null
  sourceUrl?: string | null
}
```

Then Veðurstofan feeds `ProviderMapPoint[]` now, and Vegagerðin can feed the same shape later. The UI can still label them differently by `provider`.

This is the smallest future-proof correction and avoids another provider-specific prop in `TravelAuditMap`.

### High - Status pseudo-code mixes `WindDisplayStatus` with `WeatherStatus`

References:
- v134 section 5, `VedurstofanMapPoint.status: WindDisplayStatus | 'no_data'`
- v134 section 5, `status = ... ? 'rautt' : 'gult' : 'graent'`
- v134 section 5, `statusColorMap` keyed by `graent/gult/rautt/no_data`

The plan says the map point status is `WindDisplayStatus | 'no_data'`, but the pseudo-code assigns `WeatherStatus` values: `graent`, `gult`, `rautt`.

Those are not the same domain. Existing fine-grained UI uses statuses like:

- `innan-marka`
- `nalgast-othaegindi`
- `othaegilegt`
- `nalgast-haettumork`
- `haettulegt`
- `no_data`

Recommended fix:

- For map/provider points, use `WindDisplayStatus`, not `WeatherStatus`.
- Compute it with existing classifier:

```ts
const status = classifyPointWindDisplayStatus(windMs ?? undefined, windMs !== null, effectiveThresholds)
```

- Then use:

```ts
const color = WIND_STATUS_MARKER_COLOR[status]
const meta = WIND_STATUS_UI_META[status]
```

This preserves the same severity language as MET/Yr and avoids collapsing everything into only green/yellow/red.

### High - v134 field names use `forecastTimeIso`, but Veðurstofan rows currently use `ftimeIso`

References:
- v134 section 4: `nearestRow.forecastTimeIso`
- v134 section 5: `forecastTimeIso = nearestRow?.forecastTimeIso ?? null`
- Current Veðurstofan row shape from earlier reviews: `ftimeIso`

The plan likely will not typecheck if implemented literally. Veðurstofan forecast rows use `ftimeIso`, not `forecastTimeIso`.

Recommended fix:

- Either normalize rows as part of `ProviderForecastRow` and expose `timeIso` / `forecastTimeIso`, or
- use `nearestRow.ftimeIso` wherever reading from `VedurstofanTravelLayer.points[].forecastRows`.

Prefer normalization for future providers:

```ts
type ProviderForecastRow = {
  timeIso: string
  provider: WeatherProviderKey
  windSpeedMs: number | null
  precipitationMmPerHour: number | null
  temperatureC: number | null
  windDirectionText?: string | null
  weatherText?: string | null
}
```

Then the UI never cares whether the source calls it `ftimeIso`, `timeIso`, or something else.

- Stebbi latest comment: "Veðurstofutölur eru ekki að koma í scrubberinn"

Stebbi is noticing that Veðurstofan values are not in the scrubber. v134 chooses not to implement Veðurstofan departure-slot coloring yet, which may be acceptable as a short interim, but only if the UI is honest and the plan states this clearly.

Recommended decision:

- If this patch is meant to be a small prerelease stabilization: hide the scrubber in Veðurstofan-only mode and show reference-time copy.
- If this patch is meant to satisfy "Veðurstofan in scrubber": implement a simple provider-derived scrubber now:
  - for each departure slot,
  - for each station, estimate ETA via routeFraction,
  - pick nearest forecast row,
  - classify station status,
  - slot status = worst station status,
  - selected slot drives worst/selected station.

Do not leave any MET/Yr scrubber semantics visible when `met.no` is off.

### Medium - `reference departure time` must be defined without MET/Yr optimization

References:
- v134 section 3: "Do not use `activeOutboundCandidate` for departure label... show the user-selected departure time directly."
- v134 section 5 still uses `activeOutboundCandidate` in pseudo-code.

The plan correctly says not to use MET/Yr candidate as the Veðurstofan-only departure label, but the map/status pseudo-code still uses `activeOutboundCandidate.departureIso`.

Recommended fix:

- Add a helper that is not semantically tied to MET/Yr:

```ts
const referenceDepartureIso =
  userSelectedDepartureIso
  ?? earliestDepartureAt
  ?? result.travelPlan?.requestedDepartureIso
  ?? new Date().toISOString()
```

The exact source depends on current data shape, but it should be named and treated as a neutral route/time reference, not a MET/Yr-selected "best" candidate.

Route duration and routeFraction are okay to use: they come from route geometry/Google, not weather provider data.
