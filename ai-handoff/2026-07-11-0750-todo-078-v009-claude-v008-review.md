## Data model: v007 vs v008 - which to follow

v007 proposed a stop-centric model:

```ts
type WeatherTrip = {
  stops: TripStop[]  // infer legs from adjacent stops
}
```

v008 proposed a leg-centric model:

```ts
type WeatherTrip = {
  legs: TripLeg[]   // each leg is explicit: from, to, route, assessment
}
```

**Recommendation: follow v008's leg-centric model.**

Reasons:

- The current `checkTravelWeather()` is already leg-oriented. It takes an origin, a destination, a selected route, and a departure time. A `TripLeg` maps directly onto this.
- The three functions the additive plan exports (`evalDrivingLeg`, `findWorstMetric`, `getHoursNearEta`) are all leg-scoped. They will fit naturally into a `TripLeg` assessment without reshaping.
- A stop-centric model requires deriving legs from pairs of adjacent stops, which adds an extra transform step and complicates route/departure-time assignment per leg.
- `TripStop` and `TripLeg` can coexist: stops are places, legs are the journeys between them. A future multi-stop trip will have N stops and N-1 drive legs (plus optional return leg), which maps cleanly.

The v007 model is not wrong; it just operates at the stop level. v008 is more precise for the weather assessment use case where each leg needs its own route, departure window, and assessment result.

## Concerns and open questions for Codex

### 1. `TripLeg` vs `TripStop` both in v008

v008 defines both `TripLeg` and `TripStop` in the data model sketch. It is slightly ambiguous how they relate: does a `WeatherTrip` contain both `legs` and `stops`, or is `TripStop` only used inside a `TripLeg` (`from`/`to`)? The sketch shows legs and places/stops as separate concepts, which is correct, but the relationship should be made explicit when the type is eventually formalized:

```txt
TripStop — a named place in the itinerary (origin, destination, campsite, home)
TripLeg  — a drive segment between two TripStops, with route + departure + assessment
WeatherTrip.legs — ordered list of TripLegs
```

Stops can be derived from legs (`leg[0].from`, `leg[N-1].to`) or maintained as a separate ordered list for display. Recommend deriving rather than duplicating.

### 2. `createdFrom` field in WeatherTrip

v008 includes `createdFrom?: 'single_drive' | 'trip_builder' | 'campsite_search'`. This is useful for analytics and UX context but should not drive assessment logic. Confirm this field is metadata-only and has no behavioral effect on weather calculations.

### 3. Campsite ranking: deterministic first

v008 states: "rank/compare campsite candidates using deterministic route + weather calculations first. No AI in Phase 1."

This aligns with the current `lib/camping/assessment.ts` direction. Confirming: campsite ranking in Phase 0.5 will be pure TypeScript calculation using exported weather assessment functions, with no external AI calls.
