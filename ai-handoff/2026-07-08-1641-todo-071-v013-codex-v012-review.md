## Findings

### Medium - active mode is effectively the default, so the detail list loses metrics immediately after route calculation

v012 fixes the original stale-data bug by suppressing `summaryForWindow` metrics whenever a heatmap slot is active:

- `app/auth-mvp/vedrid/FerdalagidClient.tsx:1214` to `app/auth-mvp/vedrid/FerdalagidClient.tsx:1238`

That is the right safety move for selected-slot consistency. The risk is that the app auto-selects the first outbound slot whenever a new result arrives:

- `app/auth-mvp/vedrid/FerdalagidClient.tsx:380` to `app/auth-mvp/vedrid/FerdalagidClient.tsx:386`

Because of that, `selectedCandidatePointStatuses !== undefined` is true immediately after most route calculations:

- `app/auth-mvp/vedrid/FerdalagidClient.tsx:489` to `app/auth-mvp/vedrid/FerdalagidClient.tsx:494`
- `app/auth-mvp/vedrid/FerdalagidClient.tsx:943` to `app/auth-mvp/vedrid/FerdalagidClient.tsx:949`

So the handoff's "Do NOT select a heatmap slot" localhost check is misleading: a slot is already selected by default unless Stebbi explicitly clicks the selected slot again to deselect it. In practice, `Allir spápunktarnir á leiðinni` will usually enter active mode right away and stop showing forecast time, wind, precipitation and temperature for green/yellow/red rows.

This may be an acceptable Option A tradeoff, but it should be an explicit product decision because it partially walks back the earlier TODO #71 goal of showing the richer point details across all detail cards.

Recommended choices:

1. If selected-slot consistency is now the default UX, keep v012 behavior but update localhost expectations and accept that point rows are status/ETA-only until Option B adds per-point selected-slot metrics.
2. If the rich `summaryForWindow` list should remain the default after calculation, track explicit user selection separately from the initial auto-selected slot and only enable active-mode rows after the user actually changes/selects a heatmap slot.
3. Best product result: implement Option B later, where selected candidate rows include per-point ETA, forecast time, wind, gust, precipitation and temperature from the selected slot itself.

I would not call this a code correctness blocker, but Stebbi should verify this exact behavior on localhost before release.

### Low - no targeted regression test for `RoutePointRow` active mode

The test suite is green, but v012 did not add a targeted test that locks the core regression:

- selected slot active;
- `pt.summaryForWindow` contains stale metrics/time from another window;
- rendered row uses selected-candidate ETA;
- rendered row does not show stale `summaryForWindow` forecast time, wind, precipitation or temperature;
- selected `no_data` point shows the no-data copy.

Given this bug came from two UI regions explaining different time windows, a small rendering/helper test would be valuable. If `RoutePointRow` remains local/private, consider extracting the row-state derivation to a helper and testing that.
