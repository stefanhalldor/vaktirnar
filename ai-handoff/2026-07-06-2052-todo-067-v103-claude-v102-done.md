## What was done

Full execution of the v102 plan: exhaustive route weather sampling, per-point candidate coloring on the map, precipitation threshold correction, and i18n fixes.

### 1. Precipitation threshold raised (thresholds.ts)

`cautionPrecipMmPerHour`: 1.0 → 2.0

Strict `>` comparison means exactly 2.0 mm/h is still green. 2.1+ triggers gult. Eliminates false alarms for light drizzle.

### 2. Exhaustive route weather sampling (routeSampling.ts — new file)

`lib/weather/routeSampling.ts` implements "exhaustive when cheap":

- Deduplicates Google route geometry points by 2dp grid (~1km cells)
- If unique cells <= 120: exhaustive mode (use all unique points)
- If unique cells > 120: distance_capped mode (10km spacing)
- Always appends last route point (destination)
- `forecastLat/forecastLon` use 3dp rounding (consistent with met.no cache keys)
- Returns `RouteWeatherSamplingDiagnostics` for debugging/logging

`MAX_EXHAUSTIVE_FORECAST_POINTS = 120`, `TARGET_WEATHER_POINT_SPACING_M = 10_000`

### 3. Route API uses new sampling (route.ts)

`app/api/teskeid/weather/travel/route.ts` replaced old 15-point step-based sampling with `sampleRouteWeatherPoints()`. `samplingDiagnostics` threaded through to `TravelPlan`.

### 4. Per-point statuses on candidates (travel.ts + types.ts)

`evaluateCandidate` now computes `pointStatuses: CandidatePointStatus[]` for each candidate. Each route point gets an ETA estimate based on distance fraction, then its local weather is evaluated. Only non-green points are stored (delta encoding). If all green, `pointStatuses` is `undefined`.

New types in `types.ts`:
- `CandidatePointStatus` — `{ routeIndex, status: 'gult'|'rautt'|'no_data' }`
- `RouteWeatherSamplingDiagnostics` — attached to `TravelPlan.samplingDiagnostics`
- `TravelCandidate.pointStatuses?: CandidatePointStatus[]`

### 5. TravelAuditMap per-candidate coloring (TravelAuditMap.tsx)

New prop `selectedCandidatePointStatuses?: CandidatePointStatus[]`:

- `undefined`: legacy behavior (uses server-computed `summaryForWindow`)
- `[]`: all markers green (selected slot is calm)
- Non-empty array: markers get their per-point status; `no_data` markers go grey; `isHighlighted` only for non-green points at selectedIndex

### 6. FerdalagidClient wires it up (FerdalagidClient.tsx)

`selectedCandidatePointStatuses` computed from selected heatmap slot's candidate. Passed to `TravelAuditMap`. Green-slot note (`heatmapMapShowsDefault`) removed since map now correctly shows green for green slots.

`nextCautionLine` now includes day label (e.g., "þri. 8. júl. kl. 16:00") when the next caution slot is on a different calendar day than the earliest departure.

### 7. DepartureHeatmap arrival i18n fix (DepartureHeatmap.tsx)

Hardcoded `kl.` in arrival text replaced with `tf('heatmapSlotTime', { time: ... })`. English now shows "at HH:MM" instead of "kl. HH:MM".

### 8. i18n keys (messages/is.json + messages/en.json)

- `heatmapSlotTime` added: IS `"kl. {time}"`, EN `"at {time}"`
- `nextCautionLine` updated: no longer includes "um" / "around" prefix (the time param now includes full date+time when cross-day, or just time when same-day)
