### Question 2: Should saved/recent places persist `place_id`?

This is a product decision that requires Stebbi's explicit approval before any implementation. Here is the proposed scope:

**What it fixes:**
Returning users who tap Garðabær or Þorlákshöfn from their saved/recent list will get the same Place ID routing fidelity as new selections from Google autocomplete.

**Proposed changes (requires approval before any code is written):**

| File | Change |
|---|---|
| `sql/71_saved_places_place_id.sql` | Add `place_id text` (nullable) to `weather_saved_places`. No NOT NULL, no default — existing rows have NULL and continue to work. |
| `lib/weather/savedPlaces.ts` | Add `placeId?: string` to `SavedWeatherPlace` and `SavedWeatherPlaceInput`. Update `savedPlaceToRoutePlace` to pass `placeId` through. |
| `app/api/teskeid/weather/saved-places/route.ts` | GET: include `place_id` in SELECT. POST/upsert: include `place_id` when present and valid. Normalize with same `normalizeOptionalPlaceId` pattern. |
| `components/weather/PlaceSearch.tsx` / `FerdalagidClient.tsx` | When `savePlaceBestEffort` is called, include `placeId` in the save payload if present. |
| `components/weather/RouteSelectionStep.tsx` | Saved-place buttons pass `placeId` through when the `SavedPlace` object has one. |

**Backward compatibility:**
- Existing saved-place rows have `place_id = NULL`. They continue to route by coordinates. No data loss.
- Once a user reselects a place from Google autocomplete, the row is updated (upsert on `place_key`) and gains `place_id` going forward.

**Not in scope for this change:**
- RLS policy changes (existing policy covers the new column automatically if it's part of the same table)
- Supabase type regeneration (manual types already maintained)
- Any changes to the coordinate dedupe key (`place_key`)

**Explicit confirmation needed from Stebbi:**
- Is this approved? (`Já, framkvæmdu` or similar)
- Or is this deferred? (Coordinate routing is acceptable for saved places for now)
