## 2. Proposed Nýlegt UI for all unread events on mobile

**Proposed cap: 50, with explicit overflow message.**

Personal loan trackers accumulate events slowly. Unread events only grow if the user never acks. A hard cap of 50 with an explicit label ("og X fleiri ólesin") is safe and honest.

Changes:
- `getUnreadRecentEventsForUser` fetches up to 51 rows.
- If 51 rows come back, display 50 and show: `"og X fleiri ólesin — Merktu öll lesin til að sjá eldri"`.
- If 50 or fewer, show all.
- No pagination, no lazy load. The list is scrollable as-is.

`RecentSection` already renders a scrollable list. No structural UI change needed, just a new overflow label.

No DB change needed. The existing `.limit()` call just changes from 3 to 51.

## 3. Exact `recent_events.payload` contract for loan diffs

Current type:
```ts
export interface RecentEventPayload {
  itemName?: string
}
```

Proposed addition (additive, no migration needed — JSONB is flexible):
```ts
export interface LoanFieldChange {
  field: 'item_name' | 'note' | 'loaned_at' | 'due_at'
  oldValue: string | null
  newValue: string | null
}

export interface RecentEventPayload {
  itemName?: string
  changes?: LoanFieldChange[]
}
```

Rules:
- `changes` is only present on `loan_updated` events.
- Only fields that actually changed are included in the array.
- `oldValue` and `newValue` are always string | null (dates as ISO strings).
- `itemName` in the top-level remains the current item name at the time of the event (for the label).
- No email, no note content beyond the fact it changed, no user IDs beyond actor.

**Privacy note on `note` field:** The note is free text entered by the user about their own item. It is client-visible to the event owner (same user). Including old/new note values is acceptable since the event is scoped to `user_id`. However, for Phase B we will only emit `{ field: 'note', oldValue: '[notatexti]', newValue: null }` — meaning we do include the content. Codex should flag if this is a concern.

## 4. Which mutations emit diff events in first pass

| Mutation | Event type | Diff included |
|----------|-----------|---------------|
| `updateLoan` | `loan_updated` | Yes — item_name, note, loaned_at, due_at |
| `updateLoanItemDetails` | `loan_updated` | Yes — item_name, note |
| `markReturned` | `loan_returned` | No — fact of return is the event |
| `undoReturn` | `loan_return_undone` | No |
| `createLoan` | `loan_created` | No |
| `deleteLoan` | `loan_deleted` | No |
| `performInvitationSend` | `loan_invitation_received` | No |

So diff is only needed for `loan_updated` events. This is a narrow, safe scope.

## 5. SQL/RPC changes needed for before/after values

**No SQL/RPC changes needed for Phase B.**

Approach: app-level pre-read in the server action, before the mutation RPC.

```ts
// In updateLoan, before calling update_loan RPC:
const { data: beforeData } = await admin
  .from('loan_items')
  .select('item_name, note, loaned_at, due_at')
  .eq('id', loanId)
  .maybeSingle()
```

This read uses the service_role client (same as the mutation), is authorized by the fact that `updateLoan` already passed `guardLoanAccess()`, and only reads the actor's own data. There is a theoretical race condition if another device edits the same loan in the same millisecond, but for a personal tracker this is negligible.

The read happens before the RPC. If the RPC fails, we have the old values in memory but never record them (the mutation failed). If the read fails, we fall back to `changes: undefined` (no diff in the event).

Alternative (RPC-returned diff) is cleaner under concurrency but requires changing the `update_loan` SQL function signature. This is deferred to a future phase unless Codex blocks the pre-read approach.

**`updateLoanItemDetails` uses the same pattern** — pre-read `loan_items` before calling `update_loan_item_details`.

## 6. Migration plan

**No SQL migration needed for Phase A or Phase B.**

- Phase A: Only changes `limit` parameter. Query and table are unchanged.
- Phase B: `recent_events.payload` is JSONB. Adding `changes` to the payload is purely additive. Old rows without `changes` remain valid — TypeScript treats the field as optional.

If a future phase needs a constraint on payload shape, a migration can add a DB check then.
