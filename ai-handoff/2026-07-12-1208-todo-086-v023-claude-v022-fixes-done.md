## Fixes applied

### Medium fix: invalid/error/empty station responses no longer cache as `ok`

`lib/weather/providers/vedurstofan.server.ts`

In the batch processing loop, added a guard before building or caching a payload:

```ts
if (!station.valid || station.errText !== '' || station.forecasts.length === 0) {
  console.warn(`[weather/vedurstofan] station ${station.stationId} skipped: ...`)
  continue
}
```

Stations that fail this guard are not added to `freshlyFetched`. The resolution step then falls back to stale cache (if available) or `unavailable`. This matches the correct fail-open behavior: a bad live response does not overwrite good stale cache, and does not surface as `ok`.

Also added `atimeIso: string | null` to `VedurstofanStationForecastCache` and populated it from `station.atimeIso` in `buildPayload`. Provenance metadata is now preserved in every cached payload.

### Low fix: duplicate input station IDs are deduped

At the start of `fetchVedurstofanForecastsForStations`:

```ts
const uniqueIds = [...new Set(stationIds)]
```

The rest of the function operates on `uniqueIds`. Duplicates no longer cause extra cache reads, extra batch slots, or double entries in the result map.
