## Findings

### Medium: History fallback blandar mögulega saman mörgum fetch-batches

`readVegagerdinCurrentFromHistory()` finnur nýjasta `last_fetched_at`, en seinni query-in notar:

```ts
.gte('last_fetched_at', batchCutoffIso)
```

Relevant lines:

- `lib/weather/providers/vegagerdinCurrent.server.ts:518`
- `lib/weather/providers/vegagerdinCurrent.server.ts:523`
- `lib/weather/providers/vegagerdinCurrent.server.ts:524`

Ef cron verður keyrt á 3 mínútna fresti, þá inniheldur 10 mín gluggi nokkrar fetch-batches. Þá getur fallback payload blandað saman nýjustu batch og eldri batchum, og dedupe getur haldið inni stöð sem var ekki í nýjustu payloadinu.

Þetta er ekki jafn hættulegt og gamla `measured_at` villan, en það er óþarfi að hafa þessa óvissu þegar við erum með batch-tíma.

Recommended fix:

- Nota exact batch þegar mögulegt:

```ts
.eq('last_fetched_at', newestRow.last_fetched_at)
```

eða mjög þröngt tolerance ef Claude Code hefur rök fyrir því að timestamptz string equality sé ótraust í Supabase/PostgREST.

Best v1: exact `.eq(...)`, þar sem `upsertVegagerdinHistory()` setur sama `fetchedAtIso` á allar rows í sömu batch.

### Low: SQL83 top comment er orðin röng

`sql/83_vegagerdin_measurements_history.sql` segir enn:

- lines 14-16: fallback lesi `newest measured_at batch`

En implementation les nú `last_fetched_at`. Þetta þarf að laga áður en migration er runnin, því SQL comments eru partur af production-audit fyrir migration.

Suggested wording:

```sql
--     weather_cache is missing or expired. Reads rows from the newest fetch batch
--     by last_fetched_at, no older than 24 hours.
```

### Low: Preview route comments eru cache-only þótt kóðinn noti history fallback

`app/api/teskeid/weather/vedurpuls/vegagerdin/stations/[stationId]/preview/route.ts` notar nú `readVegagerdinCurrentWithHistoryFallback()`, en commentið segir enn:

Þetta er lítið, en í þessum áfanga skiptir orðalag máli því við erum að forðast cache-only púlsgluggavilluna.

### Low: Provider fallback test ætti að assert-a query contract, ekki bara output

`lib/__tests__/vegagerdin-history.test.ts` bætir góðum tests við, en Supabase-chain mockið staðfestir ekki að seinni query-in noti exact batch eða rétta filter field. Það hefði ekki gripið að einhver breytti aftur í `measured_at` ef output fixture-ið væri óbreytt.

Add assertion around mocked query calls:

- first query uses `.gte('last_fetched_at', ...)`
- first query orders by `last_fetched_at`
- second query uses `.eq('last_fetched_at', newestRow.last_fetched_at)` if exact-batch fix is implemented
