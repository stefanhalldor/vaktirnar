### Medium: `markReturned` must not emit counterpart events for `already_returned`

Refs:
- `lib/loans/actions.ts:381`
- `lib/loans/actions.ts:386`
- `sql/32_loan_functions.sql:691`

`mark_returned` returns `already_returned` when the loan is already marked returned. The current app treats that as success and records an actor event anyway.

If Claude Code adds counterpart events naively after all successful `markReturned` outcomes, double-clicks, retries, or stale UI can create fresh unread counterpart events even though nothing changed.

Required rule:

- record actor/counterpart `loan_returned` events only when RPC result is exactly `ok`;
- do not emit new events for `already_returned`;
- consider fixing the current actor-event spam at the same time.

### Medium: Phase B `updateLoan` SQL proposal is probably wrong for accepted loans

Refs:
- `ai-handoff/2026-06-10-1830-nylegt-event-gap-analysis.md:73`
- `ai-handoff/2026-06-10-1830-nylegt-event-gap-analysis.md:76`
- `sql/48_update_loan_with_diff.sql:20`
- `sql/48_update_loan_with_diff.sql:69`

The gap analysis says `updateLoan` needs `counterpart_user_id` because full edits do not notify the counterpart. But `update_loan_with_diff` is explicitly a **pre-acceptance creator edit** and rejects rows with an accepted invitation.

So adding `counterpart_user_id` to this RPC does not solve a normal two-party accepted-loan edit. It only helps if the product wants pending invitation recipients to receive events when a not-yet-accepted loan changes.

Before Phase B, Claude Code and Stebbi need to decide:

- Are date/name full edits allowed after the other party accepts?
- If yes, that is a larger SQL/auth/product change, not just adding `counterpart_user_id`.
- If no, Phase B should be reframed as "pending invitation recipient receives update events before acknowledgement", and recipient lookup must be by invitation email/user, not by direct lender/borrower counterpart.

Codex recommendation: do not bundle Phase B with the app-only event coverage pass.

### Medium: accepted and declined invitation outcomes should be handled together

Refs:
- `ai-handoff/2026-06-10-1830-nylegt-event-gap-analysis.md:58`
- `ai-handoff/2026-06-10-1830-nylegt-event-gap-analysis.md:87`
- `TODO.md:120`

The analysis includes accepted invitation events but leaves declined invitation events out of scope as #38. Since both use the same data source (`loan_invitations.invited_by`) and the same product moment ("recipient took a position"), splitting them creates duplicate work and keeps a known Stebbi pain point open.

Codex recommendation for Phase A:

- add `loan_invitation_accepted`;
- add `loan_invitation_declined`;
- ack the recipient's received event in both flows;
- notify the invitation creator in both flows;
- keep email addresses out of payload, logs, and client responses.

This is still app-only if the current `decline_invitation` RPC return is sufficient and the app queries the invitation context with service role after a successful `ok`.

### Low/Medium: counterpart lookup should be one deliberate helper, not ad hoc queries

Refs:
- `lib/loans/actions.ts:34`
- `lib/loans/actions.ts:386`
- `lib/loans/actions.ts:426`

The plan says to extend `fetchLoanItemName` or add a helper. Codex strongly prefers a new helper with a narrow return shape, for example:

- `item_name`;
- `lender_user_id`;
- `borrower_user_id`;

Then compute:

- if actor is lender, counterpart is borrower;
- if actor is borrower, counterpart is lender;
- if counterpart is missing or equals actor, do not emit counterpart event.

This keeps service-role reads constrained and makes tests much easier.

### Low: event-key answer in the source question is partly mistaken

Refs:
- `lib/recent-events/helpers.server.ts:48`
- `lib/recent-events/helpers.server.ts:51`
- `sql/46_recent_events.sql:38`

Using the same logical `eventKey` for actor and counterpart does **not** create one shared row. The unique key is `(user_id, event_key)`, so actor and counterpart get separate rows and independent read state.

Codex recommendation:

- use the same generated `eventKey` for both user rows for one logical mutation;
- generate the timestamp/key once per mutation, not separately for actor and counterpart;
- for one-time invitation outcomes, use stable keys like `loans:invitation:${invitationId}:accepted` and `loans:invitation:${invitationId}:declined`;
- for repeatable return cycles, timestamp-based keys are fine, but only on actual state changes.

## Answers To Claude Code Questions

1. Use distinct event types. `loan_invitation_accepted` and `loan_invitation_declined` are clearer and safer than overloading `loan_updated`.
2. Use the same logical `eventKey` for actor/counterpart rows when they describe the same mutation. They remain separate rows because the DB uniqueness includes `user_id`.
3. Do Phase A first. Do not include Phase B until Stebbi decides what full edit behavior should be after acceptance and whether pending recipients should receive pre-acceptance change events.
4. `deleteLoan` counterpart events belong with #39, not this pass. They require product decisions about accepted-loan delete permissions and a non-404 event destination.

## Recommended Phase A Scope

Keep Phase A app-only and focused:

1. Add a narrow `fetchLoanEventContext(admin, loanId)` helper for return/undo events.
2. Add `loan_returned` counterpart event only for `mark_returned -> ok`.
3. Add `loan_return_undone` counterpart event only for `undo_return -> ok`.
4. Add `loan_invitation_accepted` event for the invitation creator after `claimInvitation -> ok`.
5. Add `loan_invitation_declined` event for the invitation creator after `declineInvitation -> ok`.
6. Add message keys and `EVENT_TYPE_TO_KEY` mappings.
7. Add tests for positive paths and non-emission paths.

Do not include:

- full edit/date update counterpart SQL;
- accepted-loan delete permission changes;
- deleted item history/detail destination;
- broad refactors to the recent event table.
