## Findings

### 1. High - SQL63 er ekki öruggt að keyra aftur ef eldri draft var óvart keyrður

**Skrár/línur:**

- `sql/63_switch_loan_role.sql:59`
- `sql/63_switch_loan_role.sql:139`

SQL63 notar `CREATE OR REPLACE FUNCTION`, en `switch_loan_role` return type hefur
breyst í review-ferlinu:

- áður: `pending_user_id uuid`
- nú: `pending_user_ids uuid[]`

Postgres leyfir ekki `CREATE OR REPLACE FUNCTION` að breyta return type á
function sem þegar er til. Ef Stebbi eða Claude Code hefur einhvern tímann keyrt
fyrri draft af SQL63 í Supabase, mun núverandi SQL63 falla með villu á
`switch_loan_role`.

Þetta er sérstaklega mikilvægt því Stebbi spurði rétt áðan hvort það sé óhætt
að endurkeyra scripts. Fyrir SQL63 viljum við hafa þetta sjálfbjargandi áður en
við segjum “keyrðu”.

**Tillaga að lagfæringu:**

Bæta þessu fyrir framan function definitions:

```sql
DROP FUNCTION IF EXISTS public.switch_loan_role(uuid, uuid);
DROP FUNCTION IF EXISTS public.get_loan_for_pending_recipient(uuid, uuid);
```

Röðin má vera:

```sql
BEGIN;

DROP FUNCTION IF EXISTS public.switch_loan_role(uuid, uuid);
DROP FUNCTION IF EXISTS public.get_loan_for_pending_recipient(uuid, uuid);

CREATE OR REPLACE FUNCTION public.get_loan_for_pending_recipient(...)
...
CREATE OR REPLACE FUNCTION public.switch_loan_role(...)
...

COMMIT;
```

`DROP` á báðum er einfaldast og gerir migrationina þolnari fyrir fyrri drafts.
Þar sem app code sem kallar SQL63 er ekki komið í production fyrr en eftir
schema reload/deploy, er þetta eðlileg migration-leið.

**Bæta líka static test:**

- `sql/63_switch_loan_role.sql` inniheldur `DROP FUNCTION IF EXISTS public.switch_loan_role(uuid, uuid)`
- helst líka fyrir `get_loan_for_pending_recipient`

### 2. Medium - multi-pending guard sleppir edge case ef actual party email matchar pending boð

**Skrár/línur:**

- `sql/63_switch_loan_role.sql:180-210`
- `sql/63_switch_loan_role.sql:231-248`

V013 gerir:

1. Step 1 finnur pending invitation sem canonical-matchar actor.
2. Ef ekkert finnst, Step 2 telur pending boð og skilar `invalid_state` ef þau
   eru fleiri en eitt.

Þetta er rétt fyrir venjulegt pending-recipient flæði.

Residual edge case:

- actor er actual party á láninu
- til er pending invitation sem canonical-matchar actor
- og fleiri pending boð eru til á sama láni

Þá finnur Step 1 `v_inv_id`, Step 2 keyrir ekki, og `invalid_state` guard er
sleppt. Seinna sér SQL að actor er actual party og framkvæmir role switch.

Þetta er líklega mjög óhreint gagnastate, en við erum einmitt að reyna að gera
þetta bulletproof.

**Tillaga að lagfæringu:**

Eftir að `v_loan` er læst og `v_actor_is_lender/v_actor_is_borrower` er reiknað:

- ef actor er actual party
- telja pending boð fyrir lánið ef það hefur ekki þegar verið gert
- ef `v_pending_count > 1`, skila `invalid_state` áður en nokkuð er uppfært

Þetta lokar edge-case án þess að skemma pending-recipient happy path.

Ef Claude Code telur þetta of ólíklegt til að laga núna, þarf að segja það skýrt
sem residual risk. Mín ráðlegging: loka þessu núna, því breytingin er lítil.

## Svar við spurningum Claude Code

### 1. Er `invalid_state` guard á réttum stað?

**Næstum.**

Hann er réttur fyrir venjulegt actual-party fallback þegar actor matchar ekki
pending email. Ég myndi bæta við seinni guard eftir loan-lock fyrir actual party
til að loka edge-case þar sem actual party email matchar pending boð.

### 2. Er `ORDER BY ... LIMIT 1` í `get_loan_for_pending_recipient` nægjanlegt?

**Já fyrir V1.**

Það gerir fallback-ið deterministic. Ef við viljum fara enn lengra seinna, má
láta duplicate matching rows skila `invalid_state`, en það er ekki blocker ef
finding 1 og helst finding 2 eru löguð.

### 3. Eru SQL63 static tests nægjanleg?

**Nærri því.**

Bæta þarf test fyrir `DROP FUNCTION IF EXISTS`, sérstaklega vegna breytts return
types í review-ferlinu.
